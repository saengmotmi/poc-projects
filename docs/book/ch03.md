# Chapter 3. lockfile은 왜 존재할까

지난 장에서는 tarball을 직접 내려받아 풀어 봤다. `node_modules/dayjs`는 갑자기 생기는 폴더가 아니라, 구체적인 파일 하나가 디스크 위에 풀린 결과라는 점도 확인했다. 여기까지 오면 질문이 하나 더 남는다. 이미 `package.json`에 `dayjs`가 적혀 있는데, 왜 굳이 `package-lock.json`까지 따로 있어야 할까.

이 질문은 생각보다 실무에 가깝다. 어떤 팀은 `package-lock.json`을 꼭 커밋하고, 어떤 팀은 충돌이 나면 귀찮아한다. CI에서는 `npm ci`를 쓰라고 하고, 로컬에서는 `npm install`을 쓰라고 한다. 이 차이를 이해하려면 `package.json`과 lockfile이 같은 정보를 다른 형식으로 반복한다고 보면 안 된다. 두 파일은 애초에 맡은 역할이 다르다.

이번 장에서는 같은 `dayjs`를 두 파일에서 나란히 읽어 본다. `package.json`이 말하는 범위와 lockfile이 기록하는 정확한 결과를 비교하고, 두 파일이 어긋났을 때 `npm ci`와 `npm install`이 어떻게 다르게 반응하는지도 확인한다. 여기까지 읽고 나면 "왜 lockfile이 필요한가"를 추상적인 재현성 이야기가 아니라, 실제 파일과 명령의 차이로 설명할 수 있게 된다.

## 출발점에서 이미 되는 것

이번 장도 앞선 실험 폴더를 그대로 이어받는다. 아래 상태가 이미 확보되어 있다고 가정한다.

- `pm-week1` 폴더가 있다.
- `npm install dayjs`가 한 번 성공했다.
- `package.json`과 `package-lock.json`이 있다.
- `node_modules/dayjs`가 있다.

확인은 아래 두 줄이면 충분하다.

```bash
test -f package.json
test -f package-lock.json
```

이 상태라면 바로 비교를 시작할 수 있다.

## 이번 실험의 핵심 대상

이번 장에서 붙잡을 개념은 세 가지다.

- range
- exact version
- 재현 가능성

여기서 range는 `package.json`에 적힌 "허용 범위"를 뜻한다. exact version은 실제로 선택된 구체적인 버전 하나를 뜻한다. 재현 가능성은 같은 프로젝트를 다른 환경에서 다시 설치해도 같은 결과를 만들 수 있는가에 대한 이야기다.

용어보다 먼저 눈에 들어와야 할 것은 차이의 해상도다. `package.json`은 허용 범위를 적고, lockfile은 실제로 선택된 결과를 적는다. 둘 다 `dayjs`를 말하지만, 말하는 해상도가 다르다.

## 왜 `package.json`만으로는 부족할까

팀 저장소를 새로 클론했을 때를 떠올려 보자. `package.json`만 보고 설치한다면, npm은 매번 "지금 허용 범위 안에서 어떤 버전을 골라야 하지?"를 다시 고민하게 된다. 오늘 설치할 때와 다음 달 설치할 때 레지스트리 상태가 같다는 보장도 없다. 결국 같은 프로젝트인데도 사람마다 조금씩 다른 설치 결과를 만들 가능성이 생긴다.

반대로 lockfile이 있으면, 적어도 한 번 성공했던 설치 결과를 더 높은 해상도로 다시 꺼내 쓸 수 있다. 패키지 이름만이 아니라, 정확한 버전과 tarball 주소와 무결성까지 같이 들고 있기 때문이다.

이 차이는 말로만 설명할 때보다 파일을 나란히 읽어 보면 훨씬 또렷해진다.

## 같은 `dayjs`를 두 파일에서 읽어 보기

먼저 `package.json`에는 이렇게 적혀 있다.

```json
{
  "dependencies": {
    "dayjs": "^1.11.20"
  }
}
```

반면 같은 프로젝트의 lockfile 핵심 부분은 이렇게 생겼다.

```json
{
  "packages": {
    "": {
      "dependencies": {
        "dayjs": "^1.11.20"
      }
    },
    "node_modules/dayjs": {
      "version": "1.11.20",
      "resolved": "https://registry.npmjs.org/dayjs/-/dayjs-1.11.20.tgz",
      "integrity": "sha512-YbwwqR/uYpeoP4pu043q+LTDLFBLApUP6VxRihdfNTqu4ubqMlGDLd6ErXhEgsyvY0K6nCs7nggYumAN+9uEuQ=="
    }
  }
}
```

여기서 바로 차이가 보인다.

- `package.json`은 `^1.11.20`이라는 허용 범위를 적는다.
- lockfile은 `1.11.20`이라는 실제 선택 결과를 적는다.
- 거기에 `resolved`와 `integrity`까지 붙어서, 어떤 파일을 어디서 받아야 하는지도 같이 기록한다.

즉 `package.json`만으로는 "어떤 dayjs를 원하느냐"까지는 알 수 있어도, "어떤 tarball을 받았느냐"까지는 알 수 없다. lockfile은 바로 그 빈칸을 메운다.

## 비교 스크립트로 해상도 차이 보기

두 파일이 같은 패키지를 얼마나 다르게 설명하는지 코드로 바로 확인해 보자. 아래 스크립트는 `package.json`의 range와 lockfile의 정확한 결과를 한 번에 출력한다.

`compare-lockfile.mjs`:

```js
import fs from "node:fs/promises";

// package.json은 "허용 범위"를, lockfile은 "실제 설치 결과"를 들고 있다.
const pkg = JSON.parse(await fs.readFile("./package.json", "utf8"));
const lockfile = JSON.parse(await fs.readFile("./package-lock.json", "utf8"));

const name = "dayjs";
const lockEntry = lockfile.packages[`node_modules/${name}`];

// 같은 dayjs라도 두 파일이 보여주는 정보의 해상도는 다르다.
console.log("package.json spec:", pkg.dependencies[name]);
console.log("lockfile version:", lockEntry.version);
console.log("lockfile resolved:", lockEntry.resolved);
console.log("lockfile integrity:", lockEntry.integrity);
```

실행은 이렇게 한다.

```bash
node compare-lockfile.mjs
```

실험에서 실제로 확인한 출력은 아래와 같았다.

```text
package.json spec: ^1.11.20
lockfile version: 1.11.20
lockfile resolved: https://registry.npmjs.org/dayjs/-/dayjs-1.11.20.tgz
lockfile integrity: sha512-YbwwqR/uYpeoP4pu043q+LTDLFBLApUP6VxRihdfNTqu4ubqMlGDLd6ErXhEgsyvY0K6nCs7nggYumAN+9uEuQ==
```

이 출력만으로도 차이는 충분히 보인다. `package.json`은 `dayjs`에 대해 "이 범위면 된다"라고 말하고, lockfile은 "이번엔 바로 이 파일을 받았다"라고 말한다.

## range와 exact version은 무엇이 다를까

이제 용어를 붙여 보자.

- `^1.11.20` 같은 표기는 range다.
- `1.11.20`처럼 버전 하나만 적힌 값은 exact version이다.

여기서 중요한 건 `^` 규칙을 세세하게 외우는 일이 아니다. 지금 필요한 감각은 이것뿐이다. range는 허용 범위를 말하고, exact version은 실제로 선택된 결과를 말한다.

조금 더 밀어 보면 차이는 이렇게 정리된다.

- range는 설치 전에 읽는 정보다.
- exact version은 설치가 끝난 뒤에 확인되는 정보다.
- `resolved`와 `integrity`는 exact version보다도 더 구체적이다.

즉 lockfile은 단순히 버전 하나 더 적어 두는 파일이 아니다. "이번 설치에서 실제로 무엇을 받았는가"를 다시 재현할 수 있게 만드는 파일이다.

## 두 파일이 어긋나면 무슨 일이 생길까

이제 일부러 두 파일을 어긋나게 만들어 보자. 실험은 별도 폴더에서 하는 편이 깔끔하다.

먼저 `dayjs`를 exact version으로 한 번 고정 설치한다.

```bash
mkdir pm-week3-mismatch
cd pm-week3-mismatch
npm init -y
npm install dayjs@1.11.19 --save-exact
```

이 상태에서는 `package.json`과 lockfile이 둘 다 `1.11.19`를 가리킨다. 이제 `package.json`만 손으로 바꿔 `^1.11.20`으로 올린다. lockfile은 그대로 둔다.

실험 당시 `package.json`의 핵심 부분은 이렇게 바뀌어 있었다.

```json
{
  "dependencies": {
    "dayjs": "^1.11.20"
  }
}
```

반면 lockfile은 아직 이전 결과를 들고 있었다.

```json
{
  "packages": {
    "": {
      "dependencies": {
        "dayjs": "1.11.19"
      }
    },
    "node_modules/dayjs": {
      "version": "1.11.19",
      "resolved": "https://registry.npmjs.org/dayjs/-/dayjs-1.11.19.tgz",
      "integrity": "sha512-t5EcLVS6QPBNqM2z8fakk/NKel+Xzshgt8FFKAn+qwlD1pzZWxh0nVCrvFK7ZDb6XucZeF9z8C7CBWTRIVApAw=="
    }
  }
}
```

여기서 `npm ci`를 실행하면 결과가 아주 직설적으로 나온다.

```bash
npm ci
```

실험에서 나온 핵심 오류는 아래와 같았다.

```text
npm ERR! `npm ci` can only install packages when your package.json and package-lock.json ... are in sync.
npm ERR! Invalid: lock file's dayjs@1.11.19 does not satisfy dayjs@1.11.20
```

이 메시지가 주는 정보는 분명하다. `npm ci`는 두 파일이 어긋난 상태를 허용하지 않는다. lockfile을 믿고 바로 설치하기 위해서는, 그 lockfile이 현재 `package.json`이 허용하는 범위와도 맞아야 한다.

## `npm install`과 `npm ci`는 왜 다르게 반응할까

같은 어긋난 상태에서 `npm install`을 실행하면 반응이 다르다. 이번에는 실패로 멈추기보다, `package.json`을 기준으로 다시 풀고 lockfile을 갱신한다.

실험에서 `npm install`을 한 번 더 실행한 뒤 lockfile을 다시 읽어 보면, `dayjs` 항목은 이렇게 바뀌어 있었다.

```json
{
  "packages": {
    "": {
      "dependencies": {
        "dayjs": "^1.11.20"
      }
    },
    "node_modules/dayjs": {
      "version": "1.11.20",
      "resolved": "https://registry.npmjs.org/dayjs/-/dayjs-1.11.20.tgz",
      "integrity": "sha512-YbwwqR/uYpeoP4pu043q+LTDLFBLApUP6VxRihdfNTqu4ubqMlGDLd6ErXhEgsyvY0K6nCs7nggYumAN+9uEuQ=="
    }
  }
}
```

정리하면 npm의 동작은 이렇게 읽을 수 있다.

- lockfile의 정확한 결과가 `package.json`의 범위를 만족하면, 그 결과를 다시 쓴다.
- 범위를 더 이상 만족하지 않으면, `package.json`을 기준으로 다시 설치를 풀고 lockfile을 갱신한다.
- `npm ci`는 두 파일이 정확히 맞물려 있지 않으면 멈춘다.

그래서 로컬 개발에서는 `npm install`이 lockfile을 업데이트하는 쪽으로 움직이고, CI에서는 `npm ci`로 더 엄격하게 검증하는 흐름이 자주 나온다.

## 흔한 오해

- lockfile은 `package.json`을 복사한 파일이 아니다.
- `package.json`에 버전이 적혀 있다고 해서, 실제 tarball까지 고정된 것은 아니다.
- lockfile이 있다고 해서 `package.json`이 불필요해지는 것도 아니다. 허용 범위의 기준점은 여전히 `package.json`이다.

## 이번 장에서 일부러 생략한 것

이번 장에서는 범위와 lockfile의 역할 차이에만 집중했다. 그래서 아래 문제는 잠시 뒤로 미뤘다.

- `^`, `~`, `>=` 같은 semver 문법이 각각 정확히 무엇을 뜻하는가
- 전이 의존성까지 포함하면 lockfile은 어떻게 커지는가
- `package-lock.json` 외에 `npm-shrinkwrap.json`, `yarn.lock`, `pnpm-lock.yaml`은 어떤 계약을 가지는가

지금 필요한 건 "허용 범위"와 "실제 설치 결과"가 서로 다른 층위의 정보라는 감각이다.

## 정리

이번 장의 핵심은 한 문장으로 줄일 수 있다. `package.json`은 내가 받아들일 수 있는 범위를 적고, lockfile은 내가 실제로 설치했던 결과를 적는다.

여기까지 읽은 뒤라면 아래 문장을 자기 말로 설명할 수 있어야 한다.

- range는 허용 범위다.
- exact version은 실제로 선택된 결과다.
- lockfile은 version만이 아니라 `resolved`와 `integrity`까지 기록한다.
- `npm ci`는 `package.json`과 lockfile이 어긋나면 멈춘다.
- `npm install`은 필요하면 `package.json`을 기준으로 lockfile을 갱신한다.
