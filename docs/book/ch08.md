# Chapter 8. lockfile을 읽는 설치기

지난 장에서 `pocpm.lock.json`을 쓰는 데까지 왔다. 설치 결과는 이제 텍스트 파일로 남는다. 그런데 여기서 한 가지 어색한 점이 드러난다. 기록은 남겼는데, 다음 설치에서 그 기록을 아직 쓰지 않는다. 그러면 lockfile은 좋은 로그일 수는 있어도, 아직 설치기의 기준점은 아니다.

이 장의 목표는 분명하다. write-only lockfile을 read-write lockfile로 바꾸는 것이다. 즉 설치기가 "이번에는 무얼 고를까?"를 매번 처음부터 다시 묻기보다, 이미 성공했던 해를 우선 재사용하도록 만드는 쪽으로 한 걸음 나아간다.

## 출발점에서 이미 되는 것

현재 시점의 `pocpm`은 아래까지 구현돼 있다고 본다.

- resolve, fetch, link가 동작한다.
- `pocpm.lock.json`을 기록한다.
- `.pocpm-store/tarballs`에 tarball 캐시를 남긴다.

다만 중요한 한계도 함께 있다.

- 다음 설치가 시작될 때 lockfile을 아직 읽지 않는다.
- 그래서 매번 다시 resolve를 시도한다.
- lockfile과 현재 입력이 어긋날 때의 정책도 없다.

즉 기록은 있지만, 재사용은 없는 상태다.

## 이번 실험의 핵심 대상

이번 장에서 붙잡을 것은 두 가지다.

- lockfile read
- frozen install

lockfile read는 말 그대로 기존 `pocpm.lock.json`을 설치 입력으로 다시 쓰는 기능이다. frozen install은 lockfile과 현재 입력이 어긋나면 조용히 고치지 않고 멈추는 모드다.

## 왜 write-only lockfile로는 부족할까

지금 상태에서도 설치는 된다. 그런데 똑같은 프로젝트를 두 번째 설치할 때를 생각해 보면, 이미 알고 있는 답을 또 풀고 있다는 느낌이 강하다.

- 이미 `pocpm.lock.json`에는 어떤 descriptor가 어떤 locator로 풀렸는지 적혀 있다.
- 이미 `.pocpm-store/tarballs`에는 내려받은 tarball도 있다.

그런데 설치기가 이 둘을 읽지 않고 다시 registry metadata부터 물어보면, 기록과 캐시를 남긴 보람이 반쯤 사라진다. lockfile을 쓰는 진짜 이유는 저장 그 자체가 아니라, 그 저장 결과를 다음 설치의 기준으로 재사용하는 데 있다.

## 먼저 무엇을 읽어야 할까

가장 자연스러운 출발점은 `loadProject` 단계다. 프로젝트를 로드할 때 `package.json`만 읽는 것이 아니라, 같은 디렉터리에 `pocpm.lock.json`이 있으면 함께 읽어 둔다.

아이디어는 단순하다.

```ts
type ProjectContext = {
  rootDir: string;
  rootManifest: PackageJson;
  workspacesByName: Map<string, Workspace>;
  registryUrl: string;
  storeDir: string;
  tarballStoreDir: string;
  lockfilePath: string;
  existingLockfile?: ResolutionResult;
};
```

그리고 `loadProject` 안에서 아래 helper를 추가한다.

```ts
async function loadExistingLockfile(lockfilePath: string): Promise<ResolutionResult | undefined> {
  if (!(await fileExists(lockfilePath))) {
    return undefined;
  }

  return readJsonFile<ResolutionResult>(lockfilePath);
}
```

핵심은 이것뿐이다. 프로젝트 로딩 시점에 현재 입력과 함께 "지난번 성공한 해"도 메모리에 올려 둔다.

## resolve는 어디서 lockfile을 재사용할까

그다음 분기점은 `resolveProjectDependencies`다. descriptor를 처리하기 직전에, 같은 descriptor key가 기존 lockfile에 있는지 먼저 확인한다.

모양은 대략 이렇게 간다.

```ts
function readLockedPackage(
  project: ProjectContext,
  name: string,
  range: string
): ResolvedPackage | undefined {
  const locator = project.existingLockfile?.resolutions[descriptorKey(name, range)];

  if (!locator) {
    return undefined;
  }

  return project.existingLockfile?.packages[locator];
}
```

그다음 `resolveDescriptor` 앞에서 먼저 잠긴 해를 확인한다.

```ts
const locked = readLockedPackage(project, descriptor.name, descriptor.range);

if (locked) {
  return locked;
}
```

이렇게 하면 "이미 성공했던 해가 있고, 지금 입력도 그대로라면" 굳이 registry metadata를 다시 읽지 않아도 된다.

## frozen install은 어디서 멈춰야 할까

lockfile read만으로는 아직 부족하다. 어떤 때는 설치기가 lockfile을 조용히 업데이트하지 말고, 지금 두 파일이 어긋났다고 명시적으로 실패해야 한다. 그게 frozen install이다.

인터페이스는 `InstallOptions`에 불리언 하나만 더 있어도 충분하다.

```ts
export type InstallOptions = {
  cwd: string;
  registryUrl?: string;
  frozenLockfile?: boolean;
};
```

그리고 lock된 결과를 찾지 못했을 때 아래처럼 분기한다.

```ts
const locked = readLockedPackage(project, descriptor.name, descriptor.range);

if (locked) {
  return locked;
}

if (project.frozenLockfile) {
  throw new Error(`Lockfile is missing ${descriptor.name}@${descriptor.range}`);
}

return resolveFromRegistry(project, descriptor);
```

이 정책이 주는 효과는 분명하다.

- 일반 `install`
  - lockfile이 있으면 재사용
  - 없거나 모자라면 registry에서 다시 resolve
- frozen install
  - lockfile이 지금 입력을 설명하지 못하면 실패

즉 로컬 개발과 CI의 긴장이 여기서 갈린다. 로컬에서는 보통 고치고 나아가는 편이 편하고, CI에서는 어긋남을 빨리 드러내는 편이 안전하다.

## 현재 `pocpm` 구조와 어떻게 맞물릴까

이 장의 변경이 비교적 깔끔하게 들어갈 수 있는 이유는, 현재 구조가 이미 `loadProject -> resolve -> fetch -> link`로 분리돼 있기 때문이다.

- `loadProject`
  - 기존 lockfile 읽기
- `resolve`
  - lockfile 우선 재사용 여부 결정
- `fetch`
  - 필요한 tarball만 캐시에서 찾거나 새로 받기
- `link`
  - 결과를 디스크에 반영

즉 lockfile read는 새 단계를 끼워 넣는 일이 아니라, resolve 단계의 입력을 하나 더 늘리는 일에 가깝다.

## 검증

이 장의 완료 기준은 아래처럼 잡는 편이 좋다.

1. 한 번 정상 설치를 돌려 `pocpm.lock.json`과 tarball 캐시를 만든다.
2. 같은 프로젝트에서 registry를 쓸 수 없는 조건을 만든다.
3. `--frozen-lockfile`로 설치를 다시 돌린다.
4. lockfile과 캐시만으로 설치가 끝나면 성공이다.

반대로 아래 상황에서는 실패해야 한다.

- `package.json`에 새 dependency를 추가했는데 lockfile이 아직 모를 때
- `--frozen-lockfile` 모드인데 lockfile이 descriptor를 설명하지 못할 때

즉 이 장의 검증 기준은 "잘 되면 성공"만이 아니라, "어긋나면 잘 멈추는가"까지 포함한다.

## 흔한 오해

- lockfile을 읽는다고 해서 `package.json`이 필요 없어지는 것은 아니다.
- frozen install은 더 똑똑한 install이 아니라, 더 엄격한 install이다.
- lockfile read는 fetch 단계가 아니라 resolve 단계의 문제다.

## 이번 장에서 일부러 생략한 것

이 장에서는 lockfile read의 최소 골격만 다뤘다. 아직 아래 문제는 남아 있다.

- lockfile format migration
- 부분 재설치
- workspace entry의 재사용 정책
- semver range와 lockfile 재사용의 더 복잡한 상호작용

지금 필요한 건 "기록된 해를 다음 설치의 기준으로 다시 읽는다"는 감각이다.

## 정리

이번 장의 핵심은 이렇다. lockfile은 쓰는 순간보다 읽는 순간 진짜 힘을 발휘한다. 설치기가 기존 해를 다시 사용할 수 있어야, lockfile은 단순한 기록이 아니라 설치 기준이 된다.

여기까지 읽은 뒤라면 아래 문장을 자기 말로 설명할 수 있어야 한다.

- lockfile read는 resolve 단계의 입력을 늘린다.
- frozen install은 lockfile과 현재 입력이 어긋날 때 멈추는 정책이다.
- `loadProject -> resolve -> fetch -> link` 분리 덕분에 lockfile read를 자연스럽게 넣을 수 있다.
