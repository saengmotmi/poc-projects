# Chapter 6. 전이 의존성이 등장하면 무엇이 달라질까

`dayjs`는 첫 설치기에 잘 맞는 패키지였다. dependencies가 비어 있어서, 입력 하나를 읽고 tarball 하나를 받아 풀면 끝났다. 그런데 실제 세상에서 그런 패키지만 설치하는 경우는 드물다. 대부분의 패키지는 다른 패키지에 기대고 있다.

이 순간부터 설치기의 난이도가 한 단계 올라간다. 이제는 "내가 직접 적은 의존성"만 설치해서는 부족하다. 그 패키지가 필요로 하는 child dependency까지 같이 설치해야 한다. 즉 입력이 리스트 하나에서 그래프 비슷한 모양으로 바뀐다.

## 출발점에서 이미 되는 것

앞 장까지의 설치기는 적어도 아래를 할 수 있다고 가정한다.

- `package.json`에서 direct dependency를 읽는다.
- registry metadata에서 `dist`와 `dependencies`를 꺼낸다.
- exact version 하나의 tarball을 받아 `node_modules/<name>`에 푼다.

이제 바뀌는 건 단 하나다. 패키지 하나를 풀고 끝나는 대신, 그 안의 `dependencies`를 다시 읽어 다음 설치 대상으로 밀어 넣어야 한다.

## 이번 실험의 핵심 대상

이번 장에서 핵심은 두 가지다.

- 전이 의존성
- 재귀 설치

전이 의존성이라는 말이 어렵게 들릴 수 있는데, 감각은 단순하다. 내가 직접 적지는 않았지만, 내가 설치한 패키지가 필요로 해서 함께 따라오는 의존성이다. direct dependency의 "자식"이라고 생각해도 괜찮다.

## 왜 direct dependency만으로는 부족할까

예를 들어 루트 프로젝트가 `registry-lib` 하나만 적고 있다고 하자. 그런데 `registry-lib`의 코드가 내부에서 `registry-leaf`를 `require`한다면, 설치기가 `registry-lib` 폴더만 만들어 놓고 끝나서는 안 된다. 런타임은 결국 `registry-lib/node_modules/registry-leaf`도 필요로 한다.

즉 direct dependency만 설치하는 설치기는 겉보기엔 성공해도, 실제 실행 직전에 깨질 수 있다. 이 문제는 "추가 기능"이 아니라 설치기의 본질에 가깝다.

## 이번 장에서 쓰는 작은 fixture

이 장부터는 테스트용 fixture를 하나 쓴다. 실제 `pocpm` 테스트에도 들어 있는 아주 작은 의존성 트리다.

- `registry-lib@1.0.0`
  - dependencies:
    - `registry-leaf@1.0.0`
- `registry-leaf@1.0.0`
  - dependencies 없음

코드도 단순하다.

- `registry-lib/index.js`는 `require("registry-leaf")`
- `registry-leaf/index.js`는 문자열 하나를 내보낸다

이 정도면 전이 의존성의 핵심만 보기 좋다.

## resolve 단계는 큐를 따라 넓어진다

전이 의존성이 생기면 resolve 단계는 direct dependency 하나만 보고 끝날 수 없다. 실제 `pocpm`의 [resolve.ts](/Users/ohjongtaek/Desktop/dev/poc-projects/packages/pocpm/src/resolve.ts) 핵심은 아래처럼 생겼다.

```ts
const queue: Descriptor[] = Object.entries(project.rootManifest.dependencies ?? {}).map(
  ([name, range]) => ({ name, range })
);

while (queue.length > 0) {
  const descriptor = queue.shift()!;
  const key = descriptorKey(descriptor.name, descriptor.range);

  if (resolutions[key]) {
    continue;
  }

  const pkg = await resolveDescriptor(project, descriptor);
  resolutions[key] = pkg.locatorKey;

  if (!packages[pkg.locatorKey]) {
    packages[pkg.locatorKey] = pkg;

    for (const [dependencyName, dependencyRange] of Object.entries(pkg.dependencies)) {
      queue.push({
        name: dependencyName,
        range: dependencyRange
      });
    }
  }
}
```

이 코드를 읽는 포인트는 단순하다.

1. 루트의 direct dependency를 큐에 넣는다.
2. 하나씩 꺼내서 resolve한다.
3. 그 패키지의 `dependencies`를 다시 큐에 넣는다.
4. 큐가 빌 때까지 반복한다.

즉 설치기가 direct dependency 리스트를 읽는 프로그램에서, "아직 처리하지 않은 의존성 큐"를 비워 가는 프로그램으로 바뀌는 것이다.

## link 단계는 재귀로 내려간다

resolve가 "무엇이 필요한가"를 넓게 모으는 단계라면, link는 "그 결과를 어디에 놓을까"를 실제 디스크에 적용하는 단계다. 전이 의존성이 생기면 여기서도 재귀가 필요해진다.

[link.ts](/Users/ohjongtaek/Desktop/dev/poc-projects/packages/pocpm/src/link.ts)의 핵심 부분은 아래처럼 내려간다.

```ts
async function linkPackageIntoDirectory(
  context: LinkContext,
  locatorKey: string,
  targetDir: string
): Promise<void> {
  const pkg = context.resolution.packages[locatorKey];
  const fetched = context.fetched.get(locatorKey);

  await mkdir(targetDir, { recursive: true });
  await tar.x({
    cwd: targetDir,
    file: fetched.tarballPath,
    strip: 1
  });

  await installDependenciesIntoDirectory(context, targetDir, pkg.dependencies);
}
```

이 함수가 보여주는 건 명확하다. 패키지 하나를 targetDir에 풀고 끝내지 않는다. 그다음 줄에서 `pkg.dependencies`를 들고 같은 설치 함수를 다시 호출한다. 즉 설치는 트리 아래로 내려가는 재귀 과정이 된다.

이 구조를 이해하면 왜 `node_modules/registry-lib/node_modules/registry-leaf` 같은 중첩 폴더가 생기는지도 자연스럽게 읽힌다.

## 실제 디스크 결과는 어떻게 보일까

fixture 프로젝트를 실제로 설치한 뒤 `node_modules` 일부를 보면 아래처럼 생겼다.

```text
node_modules/registry-bin/bin/registry-greet.js
node_modules/registry-bin/package.json
node_modules/registry-lib/index.js
node_modules/registry-lib/node_modules/registry-leaf/index.js
node_modules/registry-lib/node_modules/registry-leaf/package.json
node_modules/registry-lib/package.json
```

여기서 핵심은 이 부분이다.

```text
node_modules/registry-lib/node_modules/registry-leaf/package.json
```

`registry-leaf`는 루트 `package.json`에 직접 적혀 있지 않다. 그래도 설치 결과에는 분명히 존재한다. 이유는 하나다. `registry-lib`가 그 패키지를 필요로했기 때문이다.

이 장면이 바로 전이 의존성이 디스크에 남기는 흔적이다.

## 왜 현재 `pocpm`은 hoist 없이 시작할까

여기서 눈에 띄는 점이 하나 더 있다. `registry-leaf`가 루트 `node_modules`에 납작하게 깔리지 않고, `registry-lib/node_modules` 아래에 중첩되어 있다. 이건 현재 `pocpm`이 hoist 없는 설치 전략으로 출발하기 때문이다.

학습 단계에서는 이 선택이 꽤 좋다.

- 누가 누구를 필요로 했는지가 디스크에서 바로 보인다.
- Node의 모듈 탐색 규칙과 결과 구조를 연결하기 쉽다.
- 나중에 hoist나 dedupe를 붙일 때, 무엇이 바뀌는지도 비교하기 쉽다.

즉 지금의 중첩 구조는 비효율 때문이 아니라, 의도적으로 더 읽기 쉬운 결과를 남기기 위한 선택에 가깝다.

## 검증

fixture 설치가 끝난 뒤 아래 둘을 확인한다.

```bash
test -f node_modules/registry-lib/package.json
test -f node_modules/registry-lib/node_modules/registry-leaf/package.json
```

그리고 디렉터리 구조를 읽어 본다.

```bash
find node_modules -maxdepth 4 -type f | sort
```

위 명령에서 아래 경로가 보이면 핵심 검증은 통과다.

```text
node_modules/registry-lib/node_modules/registry-leaf/package.json
```

## 흔한 오해

- 전이 의존성은 "나중에 붙이는 옵션"이 아니다. direct dependency를 실제로 실행 가능하게 만드는 필수 단계다.
- resolve와 link는 같은 일이 아니다. resolve는 필요 패키지 목록을 넓히고, link는 그 결과를 디스크에 놓는다.
- 중첩 `node_modules` 구조가 보인다고 해서 실패한 것이 아니다. 현재 `pocpm`의 의도된 전략이다.

## 이번 장에서 일부러 생략한 것

이번 장에서는 그래프를 걷는 가장 단순한 구조만 봤다. 아직 아래 문제는 다루지 않는다.

- 같은 패키지가 여러 경로로 반복될 때 dedupe하는 문제
- peer dependency
- hoist 전략
- workspace dependency

지금 필요한 건 "설치 대상이 리스트에서 트리로 바뀌는 순간, resolve와 link가 각각 어떻게 넓어지는가"를 보는 일이다.

## 정리

이번 장의 핵심은 이렇다. direct dependency만 설치하는 설치기는 반쯤만 완성된 설치기다. 패키지가 의존하는 child dependency까지 따라가야 비로소 런타임이 필요한 파일 구조가 만들어진다.

여기까지 읽은 뒤라면 아래 문장을 자기 말로 설명할 수 있어야 한다.

- 전이 의존성은 direct dependency의 자식이다.
- resolve는 큐를 따라 필요한 패키지 집합을 넓힌다.
- link는 재귀로 내려가며 nested `node_modules`를 만든다.
- 현재 `pocpm`은 hoist 없는 구조를 일부러 택하고 있다.
