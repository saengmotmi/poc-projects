# Chapter 4. exact version 하나를 설치하는 첫 설치기

지난 장까지는 설치 결과를 읽는 쪽에 집중했다. 이제는 손을 바꿔야 한다. `package.json`, lockfile, tarball, `node_modules`를 읽을 줄 아는 상태에서 한 걸음 더 가면, 결국 "그 결과를 내 코드로 직접 만들어 보자"는 질문에 닿는다. 이 장은 바로 그 지점에서 시작한다.

다만 처음부터 모든 문제를 한꺼번에 끌어오면 금방 흐려진다. 그래서 첫 설치기는 일부러 작게 시작한다. 의존성은 하나만 받는다. 버전은 range가 아니라 exact version만 받는다. workspace도 없고, 전이 의존성도 아직 없다. 지금 필요한 건 설치기의 본질을 손으로 한 번 끝까지 관통해 보는 일이다.

## 출발점에서 이미 되는 것

여기까지 읽었다면 적어도 아래 감각은 이미 손에 들어와 있어야 한다.

- `package.json`은 원하는 의존성을 적는 파일이다.
- `package-lock.json`은 실제로 설치된 결과를 더 높은 해상도로 적는 파일이다.
- tarball은 패키지 파일을 묶고 압축한 배포 산출물이다.
- `node_modules/dayjs`는 설치 결과가 디스크 위에 놓인 모습이다.

이번 장에서는 새 실험 폴더에서 정확한 버전 하나만 다룬다.

```json
{
  "name": "pm-week4",
  "private": true,
  "dependencies": {
    "dayjs": "1.11.20"
  }
}
```

여기서 `1.11.20`처럼 버전 하나만 적은 이유가 중요하다. 이제 막 첫 설치기를 만들기 시작하는데, "허용 범위 안에서 어떤 버전을 고를까"까지 한꺼번에 끌어오면 문제가 두 겹이 된다. 이 장은 그중 한 겹만 다룬다. 지금은 "정확히 이 버전을 설치하라"는 입력만 받는다.

## 이번 실험의 핵심 대상

이번 장에서 붙잡을 것은 세 가지다.

- manifest 읽기
- exact version 확인
- tarball을 `node_modules`에 풀기

이 셋만 되면 패키지 매니저의 최소 골격이 보이기 시작한다. 아직 버전 선택 알고리즘도 없고 lockfile 재사용도 없지만, 적어도 "선언을 읽고, 파일을 받고, 디스크에 놓는 프로그램"이라는 뼈대는 만들어진다.

## 왜 exact version 하나부터 시작할까

패키지 매니저를 직접 만들겠다고 하면 사람 마음은 금방 커진다. `npm install react`도 되고, `workspace:*`도 되고, lockfile도 읽고, `.bin`도 만들고 싶어진다. 그런데 그렇게 출발하면 첫 성공 경험이 너무 늦어진다.

첫 설치기에서 정말 필요한 질문은 더 작다. `dayjs@1.11.20` 하나를 설치하라고 했을 때, 내 프로그램이 그 결과를 `node_modules/dayjs`로 만들어 낼 수 있는가. 이 질문에 아직 답하지 못한 상태에서는 뒤의 기능이 다 공중에 뜬다.

정확한 버전 하나만 받는 이유도 여기에 있다. 버전 선택 문제를 잠시 치워 두면, 설치기의 핵심 흐름이 선명하게 보인다.

1. `package.json`을 읽는다.
2. 설치할 이름과 버전을 꺼낸다.
3. tarball 주소를 찾는다.
4. tarball을 내려받는다.
5. `node_modules/<name>`에 압축을 푼다.

이게 바로 첫 설치기다.

## 첫 설치기 코드

아래 스크립트는 exact version 하나만 설치하는 가장 작은 설치기다. `dayjs` 하나를 설치하는 데 충분한 기능만 넣었다.

`install-one.mjs`:

```js
import fs from "node:fs/promises";
import path from "node:path";
import { promisify } from "node:util";
import { execFile } from "node:child_process";

const run = promisify(execFile);

// 1) package.json에서 의존성 하나를 읽는다.
const manifest = JSON.parse(await fs.readFile("./package.json", "utf8"));
const entries = Object.entries(manifest.dependencies ?? {});

if (entries.length !== 1) {
  throw new Error("첫 설치기는 의존성 하나만 지원한다.");
}

const [name, version] = entries[0];

// 2) range 대신 exact version만 받는다.
if (!/^\d+\.\d+\.\d+(?:[-+][0-9A-Za-z.-]+)?$/.test(version)) {
  throw new Error(`Unsupported version: ${name}@${version}`);
}

// 3) registry metadata에서 정확한 버전의 tarball 주소를 찾는다.
const metadataResponse = await fetch(`https://registry.npmjs.org/${encodeURIComponent(name)}`);
const metadata = await metadataResponse.json();
const tarballUrl = metadata.versions?.[version]?.dist?.tarball;

if (!tarballUrl) {
  throw new Error(`Missing tarball metadata for ${name}@${version}`);
}

// 4) tarball을 캐시에 저장한다.
const tarballDir = "./.pocpm-store/tarballs";
const tarballPath = path.join(tarballDir, `${name}-${version}.tgz`);
await fs.mkdir(tarballDir, { recursive: true });
const tarballBuffer = Buffer.from(await (await fetch(tarballUrl)).arrayBuffer());
await fs.writeFile(tarballPath, tarballBuffer);

// 5) node_modules/<name> 폴더를 만들고 tarball을 푼다.
const targetDir = path.join("./node_modules", name);
await fs.rm("./node_modules", { recursive: true, force: true });
await fs.mkdir(targetDir, { recursive: true });
await run("tar", [
  "-xzf",
  tarballPath,
  "-C",
  targetDir,
  "--strip-components",
  "1"
]);

console.log("installed:", path.join(targetDir, "package.json"));
```

여기서 제일 중요한 줄은 사실 `tar`를 푸는 마지막 부분보다, 위쪽의 입력 제한이다.

```js
if (entries.length !== 1) {
  throw new Error("첫 설치기는 의존성 하나만 지원한다.");
}

if (!/^\d+\.\d+\.\d+(?:[-+][0-9A-Za-z.-]+)?$/.test(version)) {
  throw new Error(`Unsupported version: ${name}@${version}`);
}
```

처음부터 기능을 많이 넣지 않는다는 건 설명을 덜 한다는 뜻이 아니다. 오히려 무엇을 일부러 잘라냈는지 분명하게 보여 줘야 한다. 이 설치기는 "의존성 하나, exact version 하나"만 받는다고 코드로 못을 박는다. 그래야 나중에 기능을 하나씩 추가할 때, 어디서 문제가 새로 생기는지도 더 분명하게 보인다.

## 실제 `pocpm` 코드에서는 어떻게 보일까

책의 실험 코드는 일부러 작게 썼지만, 실제 `pocpm`의 시작점도 모양은 거의 같다. [index.ts](/Users/ohjongtaek/Desktop/dev/poc-projects/packages/pocpm/src/index.ts)에 들어 있는 핵심 흐름은 아래 한 덩어리다.

```ts
export async function installProject(options: InstallOptions): Promise<void> {
  const project = await loadProject(options.cwd, options.registryUrl);
  const resolution = await resolveProjectDependencies(project);
  const fetched = await fetchPackages(project, resolution);
  await writeLockfile(project, resolution);
  await linkProject(project, resolution, fetched);
}
```

앞으로 장이 더 진행되면 이 함수에 들어가는 각 단계가 점점 살이 붙는다. 하지만 지금 시점에서는 이름만 읽어도 충분하다.

- `loadProject`: 입력을 읽는다.
- `resolveProjectDependencies`: 무엇을 설치할지 정한다.
- `fetchPackages`: tarball을 받는다.
- `writeLockfile`: 결과를 기록한다.
- `linkProject`: `node_modules`를 만든다.

첫 설치기의 뼈대는 이미 이 안에 다 들어 있다.

## 검증

실험 폴더에 아래 `package.json`을 둔다.

```json
{
  "name": "pm-week4",
  "private": true,
  "dependencies": {
    "dayjs": "1.11.20"
  }
}
```

그다음 아래 순서로 실행한다.

```bash
node install-one.mjs
node -e "console.log(require('./node_modules/dayjs/package.json').name)"
test -f node_modules/dayjs/package.json
```

실험에서는 두 번째 명령이 실제로 아래처럼 출력됐다.

```text
dayjs
```

이 결과가 의미하는 건 크다. 설치기는 이제 적어도 exact version 하나를 받아 실제 패키지 폴더 하나를 만들 수 있다.

## 흔한 오해

- tarball만 내려받으면 설치가 끝난 것이 아니다. 런타임이 읽을 수 있는 위치에 풀어야 한다.
- `package.json`에 의존성이 적혀 있다고 해서, 이미 설치된 것은 아니다.
- exact version만 지원하는 설치기가 쓸모없는 장난감은 아니다. 이 단계가 뒤의 기능을 붙일 기준선이 된다.

## 이번 장에서 일부러 생략한 것

이번 장의 설치기는 일부러 작다. 아직 아래 문제는 다루지 않는다.

- range 해석
- 전이 의존성
- lockfile 재사용
- `.bin` 생성
- workspace

지금 필요한 건 "패키지 하나를 실제로 디스크에 설치하는 코드"를 내 손으로 한 번 끝까지 지나가 보는 일이다.

## 정리

이번 장의 핵심은 한 문장으로 줄일 수 있다. 패키지 매니저의 첫 성공 기준은 "정확한 버전 하나를 받아 `node_modules/<name>`을 만들 수 있는가"다.

여기까지 읽은 뒤라면 아래 문장을 자기 말로 설명할 수 있어야 한다.

- 첫 설치기는 exact version 하나만 받아도 충분하다.
- `package.json`에서 설치 이름과 버전을 읽는다.
- registry metadata에서 tarball 주소를 찾는다.
- tarball을 `node_modules/<name>`에 풀면 첫 설치 결과가 생긴다.
