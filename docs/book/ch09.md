# Chapter 9. `.bin`과 script 실행은 왜 필요한가

패키지를 설치하는 일과 패키지를 실행하는 일은 생각보다 다르다. `node_modules/registry-bin` 폴더가 생겼다고 해서, 곧바로 `registry-greet` 같은 명령이 쉘에서 되는 것은 아니다. 설치기가 "실행 가능한 이름"과 "실제 파일 경로" 사이의 연결까지 만들어 줘야 비로소 개발자가 느끼는 사용 경험과 맞닿는다.

이 장에서는 그 연결을 만든다. `package.json`의 `bin` 필드를 읽고, `node_modules/.bin` 아래에 링크를 만들고, 마지막으로 `run`이 그 링크를 PATH에 잡아서 루트 스크립트를 실행하는 흐름까지 본다.

## 출발점에서 이미 되는 것

앞 장까지의 설치기는 아래 정도의 상태라고 가정한다.

- direct dependency와 전이 의존성을 설치한다.
- lockfile을 기록한다.
- tarball 캐시를 남긴다.
- `node_modules` 구조를 만든다.

하지만 아직 한계도 분명하다.

- 바이너리 이름을 쉘에서 바로 쓸 수 없다.
- 루트 `package.json`의 `scripts`를 실행할 때 의존성 바이너리를 찾을 수 없다.

즉 설치는 되는데, 개발자가 체감하는 "도구가 동작한다"까지는 아직 닿지 않은 상태다.

## 이번 실험의 핵심 대상

이번 장에서 붙잡을 것은 세 가지다.

- `bin` 필드
- `node_modules/.bin`
- PATH 주입

여기서 `bin` 필드는 패키지가 "내가 제공하는 실행 파일 이름은 이것이고, 실제 파일은 여기 있다"라고 말하는 선언이다. `.bin`은 그 선언을 프로젝트 단위로 모아 놓은 얇은 진입점 폴더다.

## 왜 `.bin`이 따로 필요할까

실제 패키지 안에는 실행 파일이 대개 깊은 경로에 들어 있다. 예를 들어 테스트 fixture의 `registry-bin`은 아래처럼 생겼다.

- 패키지 이름: `registry-bin`
- `bin` 필드:
  - `registry-greet -> bin/registry-greet.js`

즉 진짜 파일은 `node_modules/registry-bin/bin/registry-greet.js`에 있다. 이걸 매번 풀 경로로 치면서 쓰는 건 불편하고, 루트 스크립트도 매번 그 경로를 직접 적기 어렵다.

그래서 패키지 매니저는 `node_modules/.bin/registry-greet` 같은 짧은 이름을 하나 더 만든다. 이건 실제 파일을 가리키는 링크다. 개발자는 이 짧은 이름을 쓰고, 설치기는 그 이름이 어디로 연결되는지 책임진다.

## `bin` 필드를 읽어 설치 정보로 정리하기

현재 `pocpm`은 [utils.ts](/Users/ohjongtaek/Desktop/dev/poc-projects/packages/pocpm/src/utils.ts)의 `normalizeBin`으로 `bin` 필드를 정리한다.

```ts
export function normalizeBin(name: string, binField: PackageJson["bin"]): Record<string, string> {
  if (!binField) {
    return {};
  }

  if (typeof binField === "string") {
    return {
      [name.includes("/") ? name.split("/").at(-1)! : name]: binField
    };
  }

  return Object.fromEntries(Object.entries(binField).sort(([left], [right]) => left.localeCompare(right)));
}
```

이 함수가 필요한 이유는 간단하다. npm 패키지의 `bin`은 문자열일 수도 있고, 여러 엔트리를 가진 객체일 수도 있다. 설치기는 이 둘을 "이름 -> 상대 경로"라는 한 가지 모양으로 정리해 둬야 다음 단계가 편해진다.

## `.bin` 링크는 어디서 만들어질까

실제 링크 생성은 [link.ts](/Users/ohjongtaek/Desktop/dev/poc-projects/packages/pocpm/src/link.ts)의 `createBinEntries`가 맡고 있다.

```ts
async function createBinEntries(
  context: LinkContext,
  nodeModulesDir: string,
  dependencies: Record<string, string>
): Promise<void> {
  if (Object.keys(dependencies).length === 0) {
    return;
  }

  const binDir = path.join(nodeModulesDir, ".bin");
  await mkdir(binDir, { recursive: true });

  for (const [name, range] of Object.entries(dependencies)) {
    const pkg = getResolvedPackage(context, name, range);
    const packageDirOnDisk = packageDirectory(nodeModulesDir, name);

    for (const [binName, relativeBinPath] of Object.entries(pkg.bin)) {
      const binTarget = path.join(packageDirOnDisk, relativeBinPath);
      const binLink = path.join(binDir, binName);
      await rm(binLink, { recursive: true, force: true });
      await symlinkRelative(binTarget, binLink);
    }
  }
}
```

읽는 포인트는 분명하다.

1. 현재 디렉터리의 direct dependency를 돈다.
2. 각 패키지가 가진 `bin` 엔트리를 읽는다.
3. `node_modules/.bin/<binName>` 링크를 만든다.
4. 그 링크는 실제 패키지 안의 실행 파일을 가리킨다.

즉 `.bin`은 어떤 복잡한 실행 환경이 아니라, "실제 실행 파일로 가는 짧은 별칭 모음"에 가깝다.

## 실제 결과는 어떻게 보일까

fixture 설치가 끝난 뒤 루트 `.bin`을 읽어 보면 실제 결과는 아래와 같았다.

```text
BIN_ENTRIES ["registry-greet"]
BIN_LINK ../registry-bin/bin/registry-greet.js
```

이 출력이 보여 주는 건 명확하다.

- `.bin` 안에는 `registry-greet`라는 이름이 생겼다.
- 그 링크는 `../registry-bin/bin/registry-greet.js`를 가리킨다.

즉 `registry-greet`라는 명령 이름과 실제 파일 경로가 연결된 상태다.

## script 실행은 PATH를 어떻게 바꾸나

이제 남은 문제는 하나다. 루트 `package.json`의 script를 실행할 때, 방금 만든 `.bin` 링크를 쉘이 찾을 수 있게 만들어야 한다.

현재 `pocpm`의 [run.ts](/Users/ohjongtaek/Desktop/dev/poc-projects/packages/pocpm/src/run.ts)는 이 부분을 아주 직접적으로 처리한다.

```ts
const child = spawn(command, {
  cwd,
  env: {
    ...process.env,
    PATH: [path.join(cwd, "node_modules", ".bin"), process.env.PATH ?? ""].join(path.delimiter)
  },
  shell: true
});
```

핵심은 PATH 앞에 `node_modules/.bin`을 끼워 넣는 것이다. 이 한 줄 덕분에 쉘은 `registry-greet` 같은 이름을 봤을 때, 먼저 프로젝트의 `.bin` 안에서 같은 이름을 찾는다.

감각적으로는 이렇게 읽으면 된다.

- 설치 단계가 "이 이름은 이 파일을 가리킨다"를 만든다.
- 실행 단계가 "이 이름을 먼저 여기서 찾아라"를 쉘에 알려 준다.

이 두 조각이 맞물려야 `npm run dev` 같은 경험이 생긴다.

## 실제 실행 결과

fixture 프로젝트의 루트 `package.json`에는 아래 스크립트가 있었다.

```json
{
  "scripts": {
    "greet": "registry-greet"
  }
}
```

그리고 `runScript(projectDir, "greet")`를 실제로 실행했을 때 나온 결과는 아래와 같았다.

```text
RUN_STDOUT hello-from-bin
RUN_EXIT 0
```

즉 루트 스크립트 안의 `registry-greet`는 문자열일 뿐이었지만, `.bin` 링크와 PATH 주입 덕분에 실제 실행 파일로 정상 연결되었다.

이 장면이 중요하다. 패키지 매니저가 설치를 마쳤다는 말은 이제 단순히 폴더를 만들어 뒀다는 뜻이 아니다. 개발자가 쓰는 명령 이름이 실제 바이너리 파일로 이어질 수 있는 상태까지 만들었다는 뜻이다.

## 검증

fixture 설치가 끝난 뒤 아래를 확인한다.

```bash
test -L node_modules/.bin/registry-greet
node -e "const fs=require('fs'); console.log(fs.readlinkSync('node_modules/.bin/registry-greet'))"
./node_modules/.bin/registry-greet
```

핵심 기대 결과는 아래 둘이다.

- `.bin/registry-greet`가 실제 파일을 가리킨다.
- `.bin/registry-greet`를 직접 실행했을 때 `hello-from-bin`이 출력된다.

## 흔한 오해

- `.bin`은 바이너리 파일을 복사해 두는 폴더가 아니다. 보통은 링크 모음에 가깝다.
- 패키지 설치와 script 실행은 같은 문제가 아니다. `.bin`과 PATH가 이어져야 실행까지 닿는다.
- `package.json`의 `scripts`는 패키지 내부 파일 경로를 직접 모를 수 있다. 그 사이를 이어 주는 것이 `.bin`이다.

## 이번 장에서 일부러 생략한 것

이번 장에서는 실행의 가장 작은 형태만 다뤘다. 아직 아래 문제는 남아 있다.

- workspace package의 bin 노출
- platform별 shim 차이
- lifecycle script 실행 순서
- PATH 충돌과 우선순위 정책

지금 필요한 건 "설치기가 파일을 놓는 것에서 끝나지 않고, 실행 이름과 실제 파일 경로도 연결한다"는 감각이다.

## 정리

이번 장의 핵심은 이렇다. 패키지 매니저는 패키지 파일만 설치하는 도구가 아니라, 패키지가 제공하는 실행 이름까지 프로젝트 런타임 환경에 연결하는 도구다.

여기까지 읽은 뒤라면 아래 문장을 자기 말로 설명할 수 있어야 한다.

- `bin` 필드는 실행 이름과 실제 파일 경로의 매핑이다.
- `.bin`은 그 매핑을 프로젝트 단위로 모아 둔 진입점 폴더다.
- PATH 앞에 `.bin`을 넣으면 루트 script가 설치된 바이너리를 바로 실행할 수 있다.
