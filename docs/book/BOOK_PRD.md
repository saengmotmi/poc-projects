# `pocpm` 교재 PRD

## 한 줄 설명

프론트엔드 개발자가 `dayjs` 하나 설치해 보는 가장 작은 실험에서 출발해, 자기 손으로 패키지 매니저를 만들면서 npm, Yarn, pnpm, Bun의 설계를 스스로 이해하게 되는 프로젝트형 개발 서적.

## 왜 이 책이 필요한가

패키지 매니저는 프론트엔드 개발자가 매일 쓰는 도구인데도, 보통은 문제가 생겼을 때만 어렴풋이 마주칩니다. 그래서 많은 독자가 `package.json`, `lockfile`, `node_modules`, `workspace`, `hoisting`, `peer dependency` 같은 말을 서로 다른 잡지식처럼 받아들이기 쉽습니다.

이 책은 그 흐름을 바꿉니다.

독자가 먼저 설치 결과를 눈으로 보고, 그 과정을 손으로 재현하고, 결국 작은 설치기를 직접 만드는 쪽으로 안내합니다. 즉 "용어를 외우는 책"이 아니라 "직접 만들면서 이해하는 책"이어야 합니다.

## 목표 독자

이 책은 아래 독자를 전제로 합니다.

- 실무 프론트엔드 개발 경험이 있다.
- `npm install`, `yarn install`, `pnpm install`은 써봤다.
- `package.json`, lockfile, `node_modules`는 익숙하지만 내부 원리는 흐릿하다.
- CS 기초 과목을 체계적으로 배우지 않았어도 괜찮다.
- 추상 개념만 오래 설명하는 책보다, 코드와 디스크 결과를 함께 보여주는 책이 더 잘 맞는다.

## 독자에게 약속하는 것

이 책은 독자에게 아래를 약속합니다.

1. 패키지 매니저를 "무서운 인프라"가 아니라 "설치 결과를 만드는 프로그램"으로 이해하게 한다.
2. 각 장이 끝날 때마다 항상 동작하는 상태를 남긴다.
3. 새로운 개념은 항상 왜 필요한지 스토리와 함께 설명한다.
4. 구현은 toy에서 멈추지 않고, 모노레포와 self-hosting까지 닿게 한다.
5. 다른 패키지 매니저의 설계를 "남의 마법"이 아니라 "같은 문제의 다른 해법"으로 보게 한다.

## 학습 목표

책을 끝까지 읽은 독자는 아래를 할 수 있어야 한다.

- `package.json`, `package-lock.json`, `node_modules`의 역할을 구분해 설명한다.
- tarball 다운로드와 압축 해제가 설치 과정에서 어떤 의미인지 설명한다.
- exact version 하나를 설치하는 최소 설치기를 직접 구현한다.
- 전이 의존성, lockfile, `.bin`, workspace가 왜 필요한지 구현 수준에서 설명한다.
- `pocpm`의 현재 구조를 읽고, 다음 구현 우선순위를 스스로 정리한다.
- npm, Yarn, pnpm, Bun의 핵심 차이를 store/linker/compatibility/migration 관점에서 비교한다.

## 비목표

이 책은 아래를 처음부터 목표로 삼지 않는다.

- npm 전체 호환 구현
- PnP 구현
- publish 생태계 완전 대응
- private registry 인증 전체 구현
- peer dependency와 hoisting의 완전한 해법
- 엔터프라이즈 운영 도구 설명서

즉 이 책은 "완성형 패키지 매니저 만들기"가 아니라, "패키지 매니저의 핵심을 직접 만들어 이해하기"를 우선한다.

## 책의 핵심 스토리

책 전체는 하나의 스토리로 읽혀야 한다.

1. 나는 프론트엔드 개발자다.
2. 매일 패키지 매니저를 쓰지만, 내부는 잘 모른다.
3. 그래서 의존성 없는 유명 패키지 하나를 직접 설치해 본다.
4. 설치 결과를 디스크에서 읽어본다.
5. 그 결과를 손으로 재현한다.
6. 그 손작업을 코드로 자동화한다.
7. 작은 설치기를 점점 키워 workspace와 self-hosting까지 간다.
8. 그다음에야 npm, Yarn, pnpm, Bun이 왜 서로 다르게 생겼는지 보이기 시작한다.

즉 이 책은 "역사를 먼저 배우고 구현하는 책"이 아니라, "구현하면서 역사를 이해하는 책"이다.

## 파트 구성 원칙

책은 4부 구성으로 간다.

- 1부: 설치 결과를 눈으로 보는 파트
- 2부: 최소 설치기를 만드는 파트
- 3부: 모노레포와 self-hosting으로 확장하는 파트
- 4부: 다른 패키지 매니저의 설계를 비교하고 `pocpm 1.0`을 설계하는 파트

각 파트는 앞 파트의 "항상 동작하는 상태"를 기반으로만 올라간다.

## 장 구성 원칙

모든 장은 아래 순서를 따른다.

1. 이번 장이 시작될 때 항상 동작하는 상태
2. 이번 장에 새로 추가되는 핵심 개념/구현
3. 왜 이게 필요한가
4. implementation
5. 개념 정리
6. 검증
7. 이번 장을 마치고 남아야 할 감각
8. 다음 장 예고

이 구조는 단순한 형식 통일이 아니라, 독자의 불안을 줄이기 위한 계약이다.

## reader-based prose 집필 원칙

이 책은 구현자가 말하고 싶은 순서가 아니라, 독자가 궁금해할 순서로 쓴다.

필수 원칙:

- 새로운 용어는 한 절에 2-3개 이상 넣지 않는다.
- 용어를 먼저 정의하지 말고, 먼저 문제 장면을 보여준 뒤 이름 붙인다.
- 모든 개념은 디스크 결과, 코드 위치, 실행 명령 중 하나와 연결한다.
- "왜 중요한가"는 항상 독자의 실제 개발 경험으로 번역한다.
- 긴 개념 설명 뒤에는 반드시 짧은 확인 실험이나 코드를 둔다.
- 한 장 안에 "실행해 보면 바로 알 수 있는 것"을 최소 하나 넣는다.

## 구현/예제 전략

교재는 문서와 코드가 같이 가야 한다.

원칙:

- 장마다 시작 상태와 종료 상태가 있어야 한다.
- 예제 코드는 가능한 한 장 단위로 독립 실행 가능해야 한다.
- 독자는 매 장 끝에서 실제로 뭔가 동작하는 결과를 확인할 수 있어야 한다.
- `pocpm` 본체 코드와 샌드박스 예제를 구분해서 관리한다.

예제 자산은 추후 아래처럼 둘 수 있다.

- `book-examples/ch01-inspect-install`
- `book-examples/ch02-unpack-tarball`
- `book-examples/ch03-install-one`

## 편집 정책

- 한국어 네이티브의 구어체를 쓴다.
- 설명은 짧고 단정하게 쓴다.
- 추측보다 관찰을 먼저 적는다.
- "쉽다"라고 말하는 대신, 왜 막히는지 먼저 인정한다.
- 표와 리스트는 독자가 비교해야 할 때만 사용한다.
- 긴 절은 중간에 "여기서 기억할 것" 문장을 넣어 호흡을 끊는다.

## 성공 기준

이 PRD가 성공하려면 아래가 충족되어야 한다.

- 독자상과 비독자상이 분명하다.
- 책 전체의 스토리와 파트 구조가 한눈에 보인다.
- 각 장의 고정 템플릿이 별도 문서로 존재한다.
- 목차 문서만 읽어도 각 장의 질문, 산출물, 검증 포인트가 보인다.
- 이후 집필자가 바뀌어도 톤과 구조를 유지할 수 있다.
