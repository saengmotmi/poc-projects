# Chapter 2. 설치는 왜 tarball을 중심으로 돌아갈까

지난 장에서 확인한 사실은 분명했다. `import "dayjs"`는 결국 `node_modules/dayjs/dayjs.min.js`로 이어진다. 설치 결과는 눈에 보였고, 런타임이 그 결과를 실제로 읽는 경로도 따라갈 수 있었다. 이제 질문 하나가 자연스럽게 남는다. 그 파일은 원래 어디서 왔을까.

`package-lock.json`에는 이미 힌트가 들어 있다. `node_modules/dayjs` 항목 안에는 `resolved`와 `integrity`가 함께 적혀 있다. `resolved`는 실제 패키지 파일의 주소고, `integrity`는 그 파일이 기대한 바이트와 같은지 확인하는 값이다. 이 둘을 따라가면 설치 과정이 조금 더 구체적인 모양으로 보이기 시작한다.

이번 실험은 `resolved` URL에서 tarball을 직접 내려받고, 압축을 풀고, 그 결과를 `node_modules/dayjs`와 비교하는 흐름으로 진행된다. 이 실험이 끝나면 `node_modules`가 갑자기 생기는 마법의 폴더가 아니라, tarball이 디스크 위에 풀린 결과라는 감각이 조금 더 단단해진다.

## 출발점에서 이미 되는 것

실험 폴더는 지난 장에서 만든 상태를 그대로 이어받는다. 적어도 아래 상태는 이미 확보되어 있어야 한다.

- `pm-week1` 폴더가 있다.
- `npm install dayjs`가 한 번 성공했다.
- `package-lock.json`이 있다.
- `node_modules/dayjs`가 있다.

확인은 아래 명령이면 충분하다.

```bash
test -f package-lock.json
test -f node_modules/dayjs/package.json
```

이 두 줄이 통과하면, tarball을 따라갈 준비는 끝난다.

## 이번 실험의 핵심 대상

이번 실험에서 눈여겨볼 것은 세 가지다.

- `resolved`
- `integrity`
- tarball

`resolved`는 구체적인 다운로드 주소다. `integrity`는 다운로드한 파일이 lockfile이 기대한 내용과 같은지 확인하는 해시다. tarball은 실제 패키지 파일을 묶어 놓은 압축 파일이다.

여기서 registry metadata라는 말도 잠깐 스쳐 지나간다. npm registry에는 패키지에 대한 설명서 같은 JSON과 실제 tarball 파일이 함께 있다. 설명서 전체를 읽기보다, 이미 lockfile에 적혀 있는 구체적인 tarball 주소를 따라간다. 추상 설명보다 실제 파일을 먼저 보는 쪽이 지금은 더 빠르다.

## 왜 tarball까지 따라가야 할까

팀 프로젝트에서 설치 문제를 겪으면 보통 `node_modules`를 지우고 다시 설치하라는 말을 먼저 듣는다. 이 조언이 통하는 이유는 `node_modules`가 소스 저장소의 원본이 아니라, 설치 과정이 만들어낸 산출물이기 때문이다. 산출물이 흔들리면 다시 만드는 편이 빠르다.

이 감각은 tarball을 직접 내려받아 보면 훨씬 또렷해진다. `node_modules/dayjs`에 들어 있는 파일들은 누군가가 손으로 하나씩 복사해 넣은 것이 아니다. 압축 파일 하나가 내려와 풀리고, 그 결과가 설치 폴더로 놓인 것이다. 설치를 tarball 중심으로 보면, 패키지 매니저가 하는 일이 훨씬 물리적으로 느껴진다.

## lockfile에서 실제 주소를 꺼내기

먼저 `package-lock.json`에서 `dayjs` 항목을 읽는다. 짧게 보려면 아래 한 줄이면 된다.

```bash
node -e "const lock=require('./package-lock.json'); const dayjs=lock.packages['node_modules/dayjs']; console.log(dayjs.resolved); console.log(dayjs.integrity); console.log(dayjs.version);"
```

실험에서 실제로 출력된 값은 아래와 같았다.

```text
https://registry.npmjs.org/dayjs/-/dayjs-1.11.20.tgz
sha512-YbwwqR/uYpeoP4pu043q+LTDLFBLApUP6VxRihdfNTqu4ubqMlGDLd6ErXhEgsyvY0K6nCs7nggYumAN+9uEuQ==
1.11.20
```

여기서 `resolved`가 가리키는 주소가 바로 tarball이다. 다만 `.tgz`라는 확장자는 처음 보면 한 번 더 설명이 필요하다.

`tar`는 여러 파일과 폴더를 하나로 묶는 아카이브 형식이다. 감각적으로는 `.zip`과 비슷해 보이지만, 원래 목적은 "압축"보다 "묶어서 보관하기"에 더 가깝다. 이름도 `tape archive`의 줄임말인데, 예전 Unix 환경에서 자기 테이프에 파일을 차례대로 담아 옮기기 위해 쓰던 형식에서 출발했다. 그래서 `tar` 자체는 파일 경로, 권한, 디렉터리 구조를 한 덩어리로 묶는 데 집중하고, 바이트를 더 작게 줄이는 압축은 별도 도구가 맡는 방식이 자연스러웠다.

그 별도 도구가 여기서는 `gzip`이다. 즉 `dayjs-1.11.20.tgz`는

- 먼저 `tar`로 파일들을 한 덩어리로 묶고
- 그 결과에 `gzip` 압축을 한 번 더 입힌 파일

이라고 보면 된다. 그래서 `.tar.gz`와 `.tgz`는 사실상 같은 뜻으로 쓰인다. npm 문맥에서 "tarball"이라고 부를 때도, 실제로는 이런 gzip이 적용된 tar 파일을 가리키는 경우가 많다.

이제 주소의 끝이 왜 `.tgz`인지도 자연스럽게 읽힌다. 이 파일 하나에 `dayjs` 패키지의 배포 산출물이 묶여 있고, 그 묶음이 gzip으로 압축되어 있다.

## tarball 내려받기와 무결성 확인

lockfile을 읽어서 tarball을 직접 내려받으면, 설치 과정의 물리적인 모양이 바로 드러난다. 아래 스크립트는 네 가지 일을 한 번에 처리한다.

1. lockfile에서 `resolved`와 `integrity`를 읽는다.
2. tarball을 내려받아 `./artifacts/dayjs.tgz`로 저장한다.
3. 다운로드한 바이트의 `sha512`를 계산해 lockfile의 `integrity`와 비교한다.
4. 압축을 풀어 `./artifacts/unpacked/package` 폴더를 만든다.

`unpack-dayjs.mjs`:

```js
import fs from "node:fs/promises";
import path from "node:path";
import { createHash } from "node:crypto";
import { promisify } from "node:util";
import { execFile } from "node:child_process";

// child_process의 execFile을 await로 쓰기 위해 Promise 형태로 감싼다.
const run = promisify(execFile);

// lockfile에서 dayjs 항목을 읽어 실제 tarball 주소와 integrity를 꺼낸다.
const lockfile = JSON.parse(await fs.readFile("./package-lock.json", "utf8"));
const dayjs = lockfile.packages["node_modules/dayjs"];

// 실험 산출물은 ./artifacts 아래에 모아 둔다.
const artifactsDir = "./artifacts";
const tarballPath = path.join(artifactsDir, "dayjs.tgz");
const unpackDir = path.join(artifactsDir, "unpacked");

await fs.mkdir(artifactsDir, { recursive: true });

// resolved URL에서 tarball을 내려받아 그대로 파일로 저장한다.
const response = await fetch(dayjs.resolved);
const tarballBuffer = Buffer.from(await response.arrayBuffer());
await fs.writeFile(tarballPath, tarballBuffer);

// npm lockfile의 integrity 형식에 맞춰 sha512-base64 문자열을 다시 계산한다.
const downloadedIntegrity =
  "sha512-" + createHash("sha512").update(tarballBuffer).digest("base64");

// 이전 실험 흔적을 지우고, tarball을 풀 폴더를 다시 만든다.
await fs.rm(unpackDir, { recursive: true, force: true });
await fs.mkdir(unpackDir, { recursive: true });

// tar -xzf는 gzip이 적용된 tarball(.tgz)을 풀어 지정한 폴더에 놓는다.
await run("tar", ["-xzf", tarballPath, "-C", unpackDir]);

// 마지막엔 "어디서 받았고, 무결성이 맞았고, 어디에 풀렸는가"를 한 번에 확인한다.
console.log("resolved:", dayjs.resolved);
console.log("integrity (lockfile):", dayjs.integrity);
console.log("integrity (downloaded):", downloadedIntegrity);
console.log("integrity matches:", dayjs.integrity === downloadedIntegrity);
console.log("tarball path:", tarballPath);
console.log("unpacked root:", path.join(unpackDir, "package"));
```

여기서 가장 낯설 수 있는 줄은 아마 이 부분일 것이다.

```js
const downloadedIntegrity =
  "sha512-" + createHash("sha512").update(tarballBuffer).digest("base64");
```

이 한 줄은 "방금 내려받은 tarball의 내용을 요약한 지문을 다시 만든다"는 뜻에 가깝다.

- `createHash("sha512")`
  - `sha512`라는 해시 알고리즘을 쓰겠다는 뜻이다.
  - 해시는 긴 파일 내용을 고정 길이의 대표값으로 바꾸는 방식이다.
- `.update(tarballBuffer)`
  - 방금 내려받은 tarball 바이트 전체를 해시 계산기에 넣는다.
- `.digest("base64")`
  - 계산 결과를 사람이 파일에 적기 쉬운 문자열 형태로 꺼낸다.
  - 여기서는 `base64` 문자열로 바꾼다.
- `"sha512-" + ...`
  - lockfile의 `integrity` 필드는 단순한 해시값만 적지 않고, 어떤 알고리즘으로 계산했는지도 같이 적는다.
  - 그래서 앞에 `sha512-`가 붙는다.

즉 lockfile의 `integrity`는 "이 tarball은 sha512로 계산했을 때 정확히 이 지문이 나와야 한다"는 뜻이다. 방금 만든 `downloadedIntegrity`와 lockfile의 `integrity`를 비교하는 이유도 여기에 있다. 주소가 맞는 파일을 받았는지보다 한 단계 더 나아가, 받은 바이트 내용이 정확히 기대한 값과 같은지 확인하는 셈이다.

실행은 이렇게 한다.

```bash
node unpack-dayjs.mjs
```

실험에서 실제로 확인한 출력은 아래와 같았다.

```text
resolved: https://registry.npmjs.org/dayjs/-/dayjs-1.11.20.tgz
integrity (lockfile): sha512-YbwwqR/uYpeoP4pu043q+LTDLFBLApUP6VxRihdfNTqu4ubqMlGDLd6ErXhEgsyvY0K6nCs7nggYumAN+9uEuQ==
integrity (downloaded): sha512-YbwwqR/uYpeoP4pu043q+LTDLFBLApUP6VxRihdfNTqu4ubqMlGDLd6ErXhEgsyvY0K6nCs7nggYumAN+9uEuQ==
integrity matches: true
tarball path: artifacts/dayjs.tgz
unpacked root: artifacts/unpacked/package
```

여기서 제일 중요한 줄은 `integrity matches: true`다. lockfile은 단순히 "이 주소에서 받으라"만 말하지 않는다. "받은 파일의 내용이 정확히 이 해시여야 한다"까지 같이 말한다. 주소와 무결성이 같이 있어야 설치 결과를 더 믿을 수 있다.

압축을 푸는 줄도 짧게 짚고 넘어가면 좋다.

```js
await run("tar", ["-xzf", tarballPath, "-C", unpackDir]);
```

여기서 `tar`는 아카이브를 푸는 도구이고, `xzf`는 세 동작을 묶어 쓴 옵션이다.

- `x`: extract, 압축을 푼다
- `z`: gzip이 적용된 파일이라는 뜻이다
- `f`: 뒤에 오는 파일 경로를 입력 파일로 쓴다

즉 이 한 줄은 "`dayjs.tgz`를 gzip 해제하면서 tar 아카이브를 풀고, 결과를 `unpackDir` 아래에 놓는다"는 뜻이다.

## tarball 안에는 무엇이 들어 있을까

압축 파일 안을 먼저 들여다보면 구조가 더 쉽게 보인다. 아래 명령으로 tarball의 앞부분만 확인할 수 있다.

```bash
tar -tzf artifacts/dayjs.tgz | head -n 12
```

실험에서 보인 첫 항목은 아래와 같았다.

```text
package/.editorconfig
package/LICENSE
package/plugin/advancedFormat.js
package/esm/locale/af.js
package/locale/af.js
package/esm/locale/am.js
package/locale/am.js
package/esm/locale/ar-dz.js
package/locale/ar-dz.js
package/esm/locale/ar-iq.js
package/locale/ar-iq.js
package/esm/locale/ar-kw.js
```

여기서 바로 보이는 특징이 하나 있다. tarball 안의 모든 파일이 `package/` 아래에 들어 있다는 점이다. npm 패키지 tarball은 이런 형태를 자주 쓴다. 압축을 풀면 `package/`라는 루트 폴더가 생기고, 그 안에 실제 패키지 파일이 놓인다.

이 구조를 눈으로 보고 나면, `artifacts/unpacked/package`가 왜 생겼는지도 자연스럽게 이해된다.

## unpacked 폴더와 `node_modules/dayjs` 비교하기

가장 중요한 비교는 tarball을 풀어 만든 폴더와, 이미 설치되어 있는 `node_modules/dayjs`를 나란히 놓는 일이다.

먼저 차이가 있는지 확인한다.

```bash
diff -rq artifacts/unpacked/package node_modules/dayjs
```

실험에서는 출력이 비어 있었다. `diff`가 아무 말도 하지 않았다는 뜻이고, 두 디렉터리가 같은 내용을 갖고 있었다는 뜻이다.

파일 개수도 확인해보면 더 안심이 된다.

```bash
find artifacts/unpacked/package -type f | wc -l
find node_modules/dayjs -type f | wc -l
```

실제 결과는 아래와 같았다.

```text
451
451
```

여기서 얻을 수 있는 감각은 꽤 중요하다. 적어도 이번 실험에서는 `node_modules/dayjs`가 tarball을 풀어 놓은 결과와 사실상 같은 모양이었다. 물론 실제 패키지 매니저는 여기에 캐시, 링크 전략, workspace 처리, script 실행 같은 일을 더 얹을 수 있다. 그래도 바닥에 깔린 기본 동작은 꽤 단순하다. 패키지 파일을 하나 받고, 무결성을 확인하고, 압축을 풀고, 런타임이 읽을 수 있는 위치에 놓는 것이다.

## 흔한 오해

- `resolved`는 registry 전체를 가리키는 말이 아니다. 이미 구체화된 tarball 주소다.
- `integrity`는 버전 문자열의 다른 표현이 아니다. 다운로드한 파일의 바이트가 기대와 같은지 확인하는 값이다.
- `node_modules`는 프로젝트 저장소의 원본 소스가 아니라, 설치 과정이 만든 산출물이다.

## 이번 장에서 일부러 생략한 것

이번 실험은 tarball을 직접 따라가는 데 집중했다. 그래서 아래 질문은 잠시 뒤로 미뤘다.

- npm은 애초에 `resolved` URL을 어떻게 알아냈을까
- 버전 범위가 있을 때 어떤 버전을 선택하는가
- tarball을 푼 뒤 `.bin`, lifecycle script, hoisting 같은 처리는 어떻게 이어지는가

이 장면들은 뒤에서 차례로 다시 등장한다. 지금 필요한 건 "설치는 결국 구체적인 파일 하나를 디스크에 가져오는 일"이라는 감각이다.

## 정리

손으로 따라간 흐름은 짧다.

1. lockfile에서 `resolved`와 `integrity`를 읽는다.
2. `resolved` URL에서 tarball을 내려받는다.
3. tarball의 `sha512`를 계산해 `integrity`와 비교한다.
4. 압축을 풀고, 그 결과를 `node_modules/dayjs`와 비교한다.

여기까지 오면 아래 문장을 자기 말로 설명할 수 있어야 한다.

- `resolved`는 실제 tarball 주소다.
- `integrity`는 다운로드한 파일의 무결성을 확인하는 값이다.
- tarball을 풀면 설치 결과와 매우 닮은 폴더가 나온다.
- `node_modules`는 갑자기 생기는 마법의 폴더가 아니라, 설치 과정이 만든 디스크 산출물이다.
