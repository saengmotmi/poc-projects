# `pocpm` 교재 장 템플릿

이 문서는 실제 장 원고를 쓸 때 지켜야 할 고정 구조와 집필 원칙을 정의합니다. 목표는 "장마다 잘 쓴 글"이 아니라, "책 전체가 한 목소리로 읽히는 글"입니다.

## 장의 기본 골격

모든 장은 아래 순서를 기본으로 사용합니다.

## 1. 장 도입

- 독자가 지금 어디쯤 왔는지 상기시킨다.
- 이번 장의 핵심 질문을 한 문장으로 말한다.
- 추상 설명보다 문제 장면을 먼저 제시한다.

예:

- "`npm install dayjs`가 끝나자 `node_modules/dayjs`가 생겼다. 그런데 그 파일은 도대체 어떻게 내 디스크로 왔을까?"

## 2. 이번 장이 시작될 때 항상 동작하는 상태

- 이전 장까지 확보된 상태를 짧게 적는다.
- 실행 명령과 확인 포인트를 함께 적는다.
- 독자가 "지금 출발점이 어디인지" 불안해하지 않게 만든다.

형식:

- 상태 2-4개
- 확인 명령 1개 이상

## 3. 이번 장에서 새로 추가할 핵심 개념/구현

- 개념은 1-2개만 도입한다.
- 구현도 1개의 핵심 diff만 다룬다.
- 개념 목록을 먼저 길게 늘어놓지 않는다.

좋은 예:

- "이번 장에서는 `resolved` URL과 tarball이라는 두 가지만 새로 본다."

나쁜 예:

- "이번 장에서는 registry, metadata, dist-tag, semver, integrity, CAS, linker를 모두 배운다."

## 4. 왜 필요한가

- 반드시 이야기로 쓴다.
- 독자의 실제 개발 경험과 연결한다.
- "왜 중요한가"를 추상 가치 대신 실제 문제로 번역한다.

예:

- "어제는 되던 설치가 오늘은 안 되는 팀 프로젝트를 한 번이라도 겪었다면, lockfile이 왜 필요한지 이미 절반은 알고 있는 셈이다."

## 5. implementation

- 가능한 한 작은 단계로 쪼갠다.
- 한 단계마다 기대 결과를 적는다.
- 코드 블록만 던지지 말고, "왜 이 줄이 필요한지"를 짧게 설명한다.

권장 순서:

1. 실패하거나 비어 있는 시작 상태
2. 최소 코드 추가
3. 실행 명령
4. 결과 확인

## 6. 중간 개념 설명

- 구현 중간에 필요한 만큼만 개념을 설명한다.
- 구현과 분리된 긴 정의 절을 만들지 않는다.
- "지금 우리가 한 일에 이름을 붙이면 ..." 방식으로 쓴다.

예:

- "`resolved` URL에서 내려받은 `.tgz` 파일이 바로 tarball이다."

## 7. 검증

- 반드시 재현 가능한 검증 단계를 둔다.
- 독자가 스스로 확인할 수 있는 체크포인트를 적는다.
- 가능하면 `명령 -> 기대 결과` 형태로 쓴다.

예:

```bash
node install-one.mjs
test -f node_modules/dayjs/package.json
```

## 8. 흔한 오해

- 이 절은 짧아야 한다.
- 독자가 거의 반드시 헷갈릴 포인트 2-3개만 적는다.
- "이건 아직 안 한다"를 숨기지 않는다.

예:

- "지금은 range를 지원하지 않는다. 그건 의도적이다."

## 9. 이번 장에서 일부러 생략한 것

- 범위 밖 항목을 명시한다.
- 독자가 "왜 이 얘기는 안 하지?"라고 느끼기 전에 먼저 설명한다.
- 이 절은 이후 장의 예고 역할도 겸한다.

## 10. 정리

- 이번 장의 한 줄 요약
- 독자가 이제 말할 수 있어야 하는 문장 2-3개
- 다음 장으로 넘어가기 위한 감각 1개

## 11. 다음 장 예고

- 다음 장의 문제를 한 문장으로 던진다.
- cliffhanger처럼 쓰되 과장하지 않는다.

예:

- "이제 우리는 패키지 하나를 설치할 수 있다. 하지만 실제 세상은 여기서부터가 시작이다. 패키지가 또 다른 패키지를 필요로하면 어떻게 될까?"

## 집필 원칙

### reader-based prose

- 구현자의 설명 순서가 아니라 독자의 이해 순서를 따른다.
- 먼저 독자가 마주할 질문을 적고, 그다음 답한다.
- 새 용어는 문제 장면 뒤에 붙인다.

### 발화 방식

- 문단의 첫 문장은 가능하면 관찰, 장면, 코드, 파일 변화 중 하나로 시작한다.
- `이 장은`, `우리는`, `이번에는`, `~하지 않는다` 같은 운영 멘트를 본문 중심 문장으로 쓰지 않는다.
- 필자의 선택 이유보다, 독자가 지금 보게 되는 현상과 결과를 먼저 적는다.
- 부정형 문장을 써야 할 때는 거기서 멈추지 않고, 바로 뒤에 "대신 무엇을 하는가"를 긍정형으로 적는다.
- 추상 명사만 연달아 두지 말고, `package.json`, `package-lock.json`, `node_modules`, 경로, 명령, 출력 중 하나와 연결한다.

좋은 예:

- "`npm install dayjs`가 끝나자 `package-lock.json`이 생겼다."
- "`import \"dayjs\"`는 결국 `node_modules/dayjs/dayjs.min.js`로 이어진다."

피해야 할 예:

- "이번 장에서는 그것을 이해해보자."
- "우리는 아직 이 문제를 다루지 않는다."
- "그래서 이 구현이 더 자연스럽다."

### 문체

- 한국어 네이티브 구어체로 쓴다.
- 문장을 짧게 쓴다.
- "쉽다"라고 단정하지 않는다.
- 막히는 지점을 먼저 인정한다.

### 개념 밀도

- 한 절에 새 개념 2-3개를 넘기지 않는다.
- 긴 개념 절 뒤에는 반드시 코드나 디스크 결과를 둔다.
- 표는 비교가 필요할 때만 쓴다.

### 구현 설명

- 코드 블록만 보여주지 않는다.
- "무엇을 바꿨는지"보다 "왜 이 변화가 필요한지"를 먼저 적는다.
- diff가 크면, 이번 장에서 주인공인 함수만 중심에 둔다.

### 독자의 불안을 줄이는 방법

- 항상 출발 상태를 먼저 고정한다.
- "지금은 일부러 생략했다"를 자주 말해준다.
- 실패해도 괜찮은 실험이라는 메시지를 숨기지 않는다.

## 장 집필 체크리스트

- [ ] 장의 핵심 질문이 첫 화면에서 보인다.
- [ ] "항상 동작하는 상태"가 있다.
- [ ] 새 개념이 2-3개 이하로 제한되어 있다.
- [ ] "왜 필요한가"가 이야기로 쓰여 있다.
- [ ] 구현 단계가 재현 가능하다.
- [ ] 검증 명령이 있다.
- [ ] 장 운영 멘트보다 관찰과 결과가 앞에 온다.
- [ ] `~하지 않는다` 문장을 썼다면, 바로 대체 행동이 긍정형으로 이어진다.
- [ ] 일부러 생략한 범위가 적혀 있다.
- [ ] 이번 장의 한 줄 요약이 있다.
- [ ] 다음 장 예고가 있다.

## 장 파일 스캐폴드

아래는 실제 장 파일을 시작할 때 복사해서 쓸 수 있는 최소 틀입니다.

```md
# Chapter N. 제목

## 장 도입

## 출발점에서 이미 되는 것

## 이번 실험의 핵심 대상

## 왜 필요한가

## implementation

## 중간 개념 설명

## 검증

## 흔한 오해

## 이번 장에서 일부러 생략한 것

## 정리

## 다음 장 예고
```
