# Chapter 1. 패키지 하나 설치하면 실제로 무슨 일이 일어날까

프론트엔드 개발자는 패키지 매니저를 매일 쓴다. 새 프로젝트를 만들면 가장 먼저 `npm install`을 하고, 문제가 생기면 `node_modules`를 지운 뒤 다시 설치하고, 버전이 꼬이면 lockfile부터 확인한다. 익숙한 도구라는 뜻이다. 그런데 익숙하다는 말과 이해하고 있다는 말은 다르다. `npm install dayjs`를 수십 번 쳐도, 그 순간 디스크에 정확히 무슨 변화가 생기는지 차분히 본 적은 의외로 많지 않다.

빈 폴더에 패키지 하나를 설치하면, 디스크에는 바로 흔적이 남는다. `package.json`이 바뀌고, lockfile이 생기고, `node_modules` 아래에 실제 코드가 놓인다. 처음에는 이 변화부터 읽는 편이 빠르다. 그래야 나중에 `resolver`, `linker`, `store` 같은 말이 나오더라도, 공중에 뜬 설명이 아니라 지금 내 디스크에서 벌어진 변화의 이름으로 받아들일 수 있다.

이번 장의 실험은 단순하다. 빈 폴더에 `dayjs` 하나를 설치하고, 설치가 끝난 뒤 남은 파일 세 개를 읽는다.

- `package.json`
- `package-lock.json`
- `node_modules/dayjs/package.json`

초점은 `dayjs`라는 라이브러리 자체보다, 설치 전과 설치 후를 비교하면서 패키지 매니저가 남기는 흔적을 읽는 데 있다.

## 출발점에서 이미 되는 것

출발점에서 필요한 건 많지 않다. Node.js와 npm이 있고, 빈 폴더에서 `npm install dayjs`가 한 번 끝까지 돌아가면 충분하다.

- Node.js와 npm이 설치되어 있다.
- 빈 폴더에서 `npm init -y`를 실행할 수 있다.
- 그 폴더에서 `npm install dayjs`를 실행할 수 있다.

실험은 아래 명령으로 시작한다.

```bash
mkdir pm-week1
cd pm-week1
npm init -y
npm install dayjs
```

이 명령이 끝나면 적어도 아래 세 가지는 보여야 한다.

- `package.json`
- `package-lock.json`
- `node_modules/dayjs`

이 상태가 첫 기준선이 된다.

## 이번 실험의 핵심 대상

이 실험에서 먼저 붙잡을 것은 세 가지다.

- `package.json`
- `package-lock.json`
- `node_modules`

순서는 단순하다. 파일 안에 실제로 적힌 내용을 먼저 읽고, 그다음 역할에 이름을 붙인다.

## 왜 설치 결과부터 읽을까

패키지 매니저를 처음 공부할 때 흔히 드는 생각이 있다. 아마 엄청 복잡한 알고리즘부터 있을 거라는 생각이다. 틀린 말은 아니다. 실제로는 버전 선택, 전이 의존성, 재현 가능성 같은 어려운 문제가 뒤에 많이 숨어 있다. 하지만 그 세계로 곧장 들어가면 거의 항상 길을 잃는다.

팀 프로젝트에서 `git clone`을 받고 `npm install`을 했는데, 어떤 사람은 잘 되고 어떤 사람은 안 되는 장면은 흔하다. 누군가는 `package-lock.json`을 보라고 하고, 누군가는 `node_modules`를 지우라고 한다. 이런 말이 왜 나오는지 이해하려면, 설치가 디스크에 남긴 흔적부터 읽을 수 있어야 한다.

패키지 매니저 이해는 명령어 암기보다 설치 전후의 차이를 읽는 쪽에 더 가깝다.

## 설치 실행과 첫 흔적

먼저 빈 폴더를 만든다.

```bash
mkdir pm-week1
cd pm-week1
npm init -y
```

여기까지 끝나면 `package.json` 하나가 생긴다. 이 파일은 지금 막 만든 프로젝트의 기본 정보만 담고 있다. 아직 `dayjs`는 없다.

실제 설치는 한 줄이면 된다.

```bash
npm install dayjs
```

이 명령이 끝나면 폴더 구조는 전보다 확실히 복잡해진다. 우리가 확인한 실제 결과는 아래와 비슷했다.

```text
pm-week1/
  package.json
  package-lock.json
  node_modules/
    dayjs/
      package.json
      esm/
      locale/
      plugin/
```

처음 보면 딱 두 가지가 눈에 들어온다. 하나는 `package-lock.json`이 새로 생겼다는 점이다. 다른 하나는 `node_modules/dayjs`가 생겼다는 점이다. 이 두 흔적이야말로 설치의 핵심 결과다.

이제 파일별로 역할을 읽어볼 차례다.

## `package.json`은 무엇을 기록할까

설치 전에는 거의 비어 있던 `package.json`에 설치 후 생긴 가장 큰 변화는 `dependencies` 항목이다.

실험 당시 `package.json`에는 `dayjs`가 의존성으로 추가되었다. 여기서 눈여겨볼 점은 버전이 대개 정확한 숫자 하나가 아니라 범위로 기록된다는 점이다. 예를 들면 이런 식이다.

```json
{
  "dependencies": {
    "dayjs": "^1.11.20"
  }
}
```

이 대목에서 먼저 보이는 사실은 하나다. `package.json`은 “내 프로젝트가 무엇을 원하느냐”를 적는 파일이라는 점이다. 다시 말해, 이 파일은 주문서에 가깝다.

이 비유가 중요한 이유가 있다. 주문서에는 보통 원하는 품목과 조건이 적히지, 실제로 배송된 물건 하나하나의 위치까지 적히지는 않는다. `package.json`도 비슷하다. “나는 `dayjs`를 쓰고 싶다”는 사실은 적지만, 설치 결과 전체를 그대로 복사해 두지는 않는다.

`package.json`은 설치의 출발점이지, 설치 결과 전체 그 자체는 아니다.

## `package-lock.json`은 무엇을 고정할까

같은 폴더 안의 `package-lock.json`은 더 낯설게 보일 수 있다. 사람이 손으로 쓰기 위한 파일처럼 보이지도 않고, 정보도 장황하다. 대신 설치 결과를 기록하는 정보는 훨씬 더 많이 담고 있다. 이 파일은 사람이 주문하는 언어보다, 설치 결과를 기록하는 언어에 가깝다.

실험 당시 `package-lock.json`의 핵심 부분은 아래와 같았다.

```json
{
  "name": "pocpm-dayjs-WRcj9D",
  "lockfileVersion": 3,
  "requires": true,
  "packages": {
    "": {
      "dependencies": {
        "dayjs": "^1.11.20"
      }
    },
    "node_modules/dayjs": {
      "version": "1.11.20",
      "resolved": "https://registry.npmjs.org/dayjs/-/dayjs-1.11.20.tgz",
      "integrity": "sha512-YbwwqR/uYpeoP4pu043q+LTDLFBLApUP6VxRihdfNTqu4ubqMlGDLd6ErXhEgsyvY0K6nCs7nggYumAN+9uEuQ=="
    }
  }
}
```

여기서 바로 눈에 들어와야 하는 건 `packages` 객체다. 이 안에는 두 종류의 항목이 있다.

- `""`: 루트 프로젝트
- `"node_modules/dayjs"`: 실제로 설치된 패키지

즉 lockfile은 “내 프로젝트가 무엇을 원했는가”와 “그 결과로 실제 무엇이 설치되었는가”를 한 파일 안에 같이 기록한다.

특히 `"node_modules/dayjs"` 항목은 매우 중요하다. 여기에는 세 가지가 들어 있다.

- `version`
- `resolved`
- `integrity`

`version`은 실제로 선택된 정확한 버전이다.  
`resolved`는 어디서 받아왔는지 알려주는 주소다.  
`integrity`는 받은 파일이 기대한 것과 같은지 확인하기 위한 해시다.

두 파일은 같은 말을 반복하지 않는다. `package.json`이 주문서라면, `package-lock.json`은 이번에 실제로 배송된 물건의 내역서에 가깝다. 팀 프로젝트에서 이 파일을 같이 커밋하는 이유도 여기에 있다. 모두가 같은 설치 결과를 다시 재현해야 하기 때문이다.

## `node_modules/dayjs`는 최종 산출물이다

마지막으로 `node_modules/dayjs/package.json`을 보면, 앞의 두 파일과 성격이 다르다는 점이 바로 드러난다. 앞의 두 파일이 내 프로젝트의 관점에서 설치를 설명했다면, 이 파일은 설치된 패키지 자신의 관점에서 자신을 설명한다.

실험에서 확인한 핵심 필드는 아래와 같았다.

```json
{
  "name": "dayjs",
  "version": "1.11.20",
  "main": "dayjs.min.js",
  "types": "index.d.ts"
}
```

세 파일의 관계가 여기서 조금 선명해진다.

- 내 프로젝트는 `package.json`에 `dayjs`를 원한다고 적었다.
- 설치 과정은 `package-lock.json`에 실제 선택된 결과를 기록했다.
- 최종적으로 `node_modules/dayjs`라는 폴더가 생겼다.

다시 말해, 우리가 실제로 사용하는 코드는 결국 `node_modules` 안에 놓인다. 설치가 끝났다는 말은, 이 폴더 구조가 런타임이 이해할 수 있는 형태로 준비되었다는 뜻이다.

이 지점이 중요하다. 패키지 매니저를 이해할 때 많은 사람이 명령어에만 집중한다. 하지만 명령어는 입구일 뿐이다. 진짜 결과물은 디스크에 남는다. 그리고 그 결과물의 대표가 바로 `node_modules`다.

## `import "dayjs"`의 도착 지점

이쯤 되면 궁금증 하나가 남는다. `node_modules/dayjs`가 생겼다는 건 알겠다. 그런데 실제 코드에서 `import dayjs from "dayjs"`라고 썼을 때, Node는 이걸 어떻게 찾아갈까.

설치 결과는 파일 몇 개가 늘어나는 데서 끝나지 않는다. 런타임이 실제로 도착할 수 있는 경로까지 연결되어야 비로소 의미가 생긴다. 다시 말해, 패키지 매니저가 만드는 최종 산출물은 단순한 보관용 폴더가 아니라, 실제 `import`가 도착할 수 있는 경로다.

`import "dayjs"`가 실제 파일 경로로 이어지는 장면이 보이면, `node_modules`가 왜 중요한지도 한 번에 이해된다. 설치된 패키지는 저장만 되는 것이 아니라, 런타임에서 실제로 읽혀야 하기 때문이다.

`trace-dayjs.mjs` 파일에 아래 내용을 넣는다.

```js
import fs from "node:fs/promises";
import path from "node:path";
import { createRequire } from "node:module";
import dayjs from "dayjs";

const require = createRequire(import.meta.url);

const resolvedPath = require.resolve("dayjs");
const packageJsonPath = require.resolve("dayjs/package.json");
const packageJson = JSON.parse(await fs.readFile(packageJsonPath, "utf8"));

console.log("import 'dayjs' resolves to:", resolvedPath);
console.log("package.json path:", packageJsonPath);
console.log("package main:", packageJson.main);
console.log(
  "main absolute path:",
  path.join(path.dirname(packageJsonPath), packageJson.main)
);
console.log("dayjs format result:", dayjs("2026-04-21").format("YYYY-MM-DD"));
```

실행은 이렇게 한다.

```bash
node trace-dayjs.mjs
```

이 실험에서 확인한 실제 경로는 아래와 같았다.

```text
/private/tmp/pocpm-dayjs-WRcj9D/node_modules/dayjs/dayjs.min.js
```

그리고 앞에서 확인했던 `node_modules/dayjs/package.json`에는 이런 정보가 있었다.

```json
{
  "name": "dayjs",
  "version": "1.11.20",
  "main": "dayjs.min.js",
  "types": "index.d.ts"
}
```

경로를 따라가면 연결이 단순해진다.

1. 코드에서는 `dayjs`라는 패키지 이름만 쓴다.
2. Node는 현재 프로젝트를 기준으로 `node_modules/dayjs`를 찾는다.
3. 그 폴더 안의 `package.json`을 읽는다.
4. 이번 실험에서는 `main`이 `dayjs.min.js`를 가리키고 있으므로, 결국 `node_modules/dayjs/dayjs.min.js`를 불러온다.

지금 단계에서 필요한 건 규칙 암기보다 연결 감각이다. 패키지 이름으로 적은 import가, 설치 결과 폴더 안의 실제 파일 경로로 이어진다는 사실이 먼저 손에 잡혀야 한다. 이 연결이 보이면 `node_modules`가 단순 캐시 폴더가 아니라는 점도 자연스럽게 따라온다.

즉, 패키지 매니저가 설치를 끝냈다는 말은 단순히 `dayjs` 폴더가 생겼다는 뜻이 아니다. 그 폴더 안의 메타데이터와 엔트리 파일이 런타임의 탐색 규칙과 맞물려, 실제 `import`가 성공할 수 있는 상태가 되었다는 뜻이다.

이 작은 스크립트는 패키지 매니저 자체는 아니지만, 설치 결과와 런타임 사이의 연결을 선명하게 보여준다. 뒤에서 설치기를 직접 만들게 되면 같은 질문이 다시 돌아온다. “내가 만든 `node_modules` 구조는 정말 런타임이 찾을 수 있는 모양인가?”라는 질문이다.

## 핵심 정리

여기까지 왔으면 세 파일의 역할을 이렇게 구분할 수 있어야 한다.

- `package.json`: 내가 무엇을 원한다고 적는 파일
- `package-lock.json`: 이번에 실제로 무엇이 설치되었는지 기록하는 파일
- `node_modules`: 설치 결과가 실제로 놓이는 폴더

버전 선택은 `package.json`에서 시작된다. 재현 가능성은 lockfile과 연결된다. 실제 실행은 `node_modules`와 연결된다. 지금은 세 갈래가 단순해 보이지만, 패키지 매니저의 복잡한 문제들은 대부분 이 셋 사이의 긴장에서 생긴다.

## 검증

검증은 아래 순서면 충분하다.

먼저 설치 결과가 존재해야 한다.

```bash
test -f package.json
test -f package-lock.json
test -f node_modules/dayjs/package.json
```

다음으로 import tracing script가 동작해야 한다.

```bash
node trace-dayjs.mjs
```

이때 적어도 아래 정보가 출력되면 된다.

- `import 'dayjs' resolves to:` 뒤에 실제 파일 경로
- `package.json path`
- `package main`
- `dayjs format result: 2026-04-21`

출력값을 외우는 것이 목적은 아니다. 패키지 이름으로 쓴 import가 실제로 `node_modules` 안의 어떤 파일로 이어지는지 스스로 찾을 수 있으면 된다.

## 이번 실험이 남기는 기준선

첫 장에서 확인한 사실은 많지 않다. 대신 기준선은 분명하다. 패키지 매니저는 명령어보다 설치 결과로 읽히기 시작한다. `package.json`, lockfile, `node_modules`가 서로 다른 역할을 맡고 있고, `import "dayjs"`는 결국 `node_modules/dayjs/dayjs.min.js`로 이어진다.

여기까지 읽은 뒤라면, 적어도 아래 문장은 자기 말로 설명할 수 있어야 한다.

- `package.json`은 주문서에 가깝다.
- `package-lock.json`은 이번 설치 결과를 기록한 파일이다.
- `node_modules`는 최종 산출물이다.
- 이번 실험에서 `import "dayjs"`는 결국 `node_modules/dayjs/dayjs.min.js`로 이어진다.
- 패키지 매니저를 이해하려면 먼저 설치 결과를 디스크에서 읽을 수 있어야 한다.

다음에는 설치 결과가 실제로 어디서 왔는지 따라가게 된다. `package-lock.json`의 `resolved` URL은 단순한 문자열이 아니다. 그 주소를 따라가면 tarball과 설치 과정의 다음 층이 열린다.
