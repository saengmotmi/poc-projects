# Chapter 5. 패키지 메타데이터는 어떻게 설치 결정을 돕나

지난 장의 첫 설치기는 실제로 `dayjs@1.11.20`을 설치했다. 그런데 그 코드를 다시 보면 한 줄이 이상하게 중요해 보인다.

```js
const tarballUrl = metadata.versions?.[version]?.dist?.tarball;
```

설치기는 tarball을 받아서 푼다. 그렇다면 tarball URL은 어디서 오는 걸까. `package.json`에는 `dayjs`와 `1.11.20`만 적혀 있을 뿐인데, 어떻게 `https://registry.npmjs.org/dayjs/-/dayjs-1.11.20.tgz` 같은 구체적인 주소를 알게 되는 걸까.

이번 장은 바로 그 질문을 다룬다. 패키지 매니저는 보통 registry metadata라는 JSON을 먼저 읽고, 그 안에서 설치에 필요한 최소 정보만 골라낸다. 지금 단계에서 정말 필요한 정보는 많지 않다. 버전, `dist.tarball`, `dist.integrity`, 그리고 `dependencies` 정도면 충분하다.

## 출발점에서 이미 되는 것

앞 장의 exact version 설치기가 있다는 가정에서 출발한다.

- `package.json`에서 `dayjs: "1.11.20"`을 읽을 수 있다.
- tarball을 내려받아 `node_modules/dayjs`로 풀 수 있다.
- 설치기의 흐름이 `읽기 -> 받기 -> 풀기`라는 점을 알고 있다.

이번 장에서는 그중 가운데 단계, 정확히는 "무엇을 받아야 하는지 알아내는 단계"만 확대해서 본다.

## 이번 실험의 핵심 대상

이번 장에서 붙잡을 것은 네 가지다.

- registry metadata
- `version`
- `dist`
- `dependencies`

여기서 `dist`는 "배포 산출물에 대한 정보"라고 읽으면 된다. tarball URL과 integrity가 이 안에 들어 있다. 그리고 `dependencies`는 다음 장에서 전이 의존성으로 이어지는 통로가 된다.

## 왜 metadata를 먼저 읽어야 할까

설치기는 `dayjs@1.11.20`이라는 이름과 버전만 가지고 출발한다. 그런데 실제로 받아야 할 것은 tarball이라는 파일이다. 즉 설치기는 이름과 버전에서 바로 tarball로 점프하지 않는다. 그 사이에 "이 버전의 배포 정보가 무엇인가"를 알려 주는 설명서가 하나 더 필요하다.

그 설명서가 registry metadata다.

이 설명서를 먼저 읽는 이유는 분명하다.

- tarball URL을 알아야 한다.
- integrity를 알아야 한다.
- 다음 장에서 child dependency를 따라가려면 `dependencies`도 알아야 한다.

즉 metadata는 다운로드 그 자체가 아니라, 다운로드와 다음 결정을 위한 입력이다.

## metadata에서 실제로 무엇을 꺼내나

아래 스크립트는 `dayjs@1.11.20`에 대한 metadata에서 설치에 필요한 필드만 골라낸다.

`resolve-one.mjs`:

```js
const name = "dayjs";
const range = "1.11.20";

const response = await fetch(`https://registry.npmjs.org/${encodeURIComponent(name)}`);
const metadata = await response.json();
const versionMetadata = metadata.versions?.[range];

if (!versionMetadata?.dist?.tarball) {
  throw new Error("Missing tarball metadata");
}

console.log(
  JSON.stringify(
    {
      name,
      version: versionMetadata.version,
      dependencies: versionMetadata.dependencies ?? {},
      dist: {
        tarball: versionMetadata.dist.tarball,
        integrity: versionMetadata.dist.integrity
      }
    },
    null,
    2
  )
);
```

실행은 이렇게 한다.

```bash
node resolve-one.mjs
```

실험에서 실제로 나온 출력은 아래와 같았다.

```json
{
  "name": "dayjs",
  "version": "1.11.20",
  "dependencies": {},
  "dist": {
    "tarball": "https://registry.npmjs.org/dayjs/-/dayjs-1.11.20.tgz",
    "integrity": "sha512-YbwwqR/uYpeoP4pu043q+LTDLFBLApUP6VxRihdfNTqu4ubqMlGDLd6ErXhEgsyvY0K6nCs7nggYumAN+9uEuQ=="
  }
}
```

이 출력은 작지만, 설치기가 다음에 무엇을 해야 하는지 거의 다 알려 준다.

- 받아야 할 버전은 `1.11.20`
- 받을 파일은 `dist.tarball`
- 그 파일의 무결성 기준은 `dist.integrity`
- 다음에 따라갈 child dependency는 `dependencies`

`dayjs`는 dependencies가 비어 있어서 이 장에서는 딱 적당하다. 정보 구조는 다 보이는데, 다음 장에서 다룰 재귀 문제는 아직 섞이지 않는다.

## exact version만 지원할 때 코드가 단순해지는 이유

지금 `pocpm`이 exact version만 받는다는 점은 metadata 처리 방식에도 그대로 드러난다. [resolve.ts](/Users/ohjongtaek/Desktop/dev/poc-projects/packages/pocpm/src/resolve.ts)의 핵심 부분은 아래처럼 생겼다.

```ts
if (!isExactVersion(descriptor.range)) {
  throw new Error(`Unsupported dependency range: ${descriptor.name}@${descriptor.range}`);
}

const response = await fetch(`${project.registryUrl}/${encodeURIComponent(descriptor.name)}`);
const metadata = await response.json();
const versionMetadata = metadata.versions?.[descriptor.range];

if (!versionMetadata?.dist?.tarball) {
  throw new Error(`Missing metadata for ${descriptor.name}@${descriptor.range}`);
}
```

이 코드는 버전 선택 문제를 거의 하지 않는다. 왜냐하면 exact version을 받는 순간 선택이 끝나 있기 때문이다. `descriptor.range`가 `1.11.20`이라면, metadata의 `versions["1.11.20"]`만 꺼내면 된다.

그래서 지금은 metadata를 "검색"한다기보다 "주소를 찾아간다"는 감각이 더 가깝다. 이 단순함이 중요하다. 아직 설치기가 작을 때는, metadata 구조를 읽는 문제와 version solving 문제를 섞지 않는 편이 훨씬 낫다.

## metadata가 결국 만드는 설치용 레코드

실제 `pocpm`은 metadata를 그대로 쓰지 않고, 설치에 필요한 필드만 모아 `ResolvedPackage`라는 형태로 정리한다. [types.ts](/Users/ohjongtaek/Desktop/dev/poc-projects/packages/pocpm/src/types.ts)의 핵심 타입은 이렇다.

```ts
export type ResolvedPackage = {
  locatorKey: string;
  name: string;
  version: string;
  reference: string;
  source: "npm" | "workspace";
  dependencies: Record<string, string>;
  bin: Record<string, string>;
  scripts: Record<string, string>;
  dist?: {
    tarball: string;
    integrity?: string;
  };
  workspaceDir?: string;
};
```

이 타입을 눈여겨볼 필요가 있다. registry에서 오는 JSON은 원래 더 크고 지저분할 수 있다. 설치기는 그 안에서 지금 필요한 정보만 뽑아 자기 도메인 모델로 다시 정리한다. 이 단계가 있어야 뒤의 `fetch`, `link`, `run`이 registry 구조에 직접 묶이지 않는다.

지금 시점에서는 이 정도만 기억하면 충분하다.

- metadata는 외부 세계의 설명서다.
- `ResolvedPackage`는 설치기 내부에서 쓰는 정리된 레코드다.

## 검증

아래 명령을 실행한다.

```bash
node resolve-one.mjs
```

그리고 적어도 아래 항목이 출력되는지 확인한다.

- `"version": "1.11.20"`
- `"dependencies": {}`
- `dist.tarball`
- `dist.integrity`

여기서 중요한 건 JSON 전체를 외우는 일이 아니다. 설치기가 다음 단계로 넘어가기 위해 꼭 필요한 정보가 정확히 무엇인지 구분할 수 있어야 한다는 점이다.

## 흔한 오해

- metadata 전체가 다 설치기에 필요한 것은 아니다.
- `dist.tarball`을 찾았다고 해서 설치가 끝난 것은 아니다. 그건 아직 다운로드 정보일 뿐이다.
- `dependencies`가 비어 있는 패키지 하나를 본 뒤, 모든 패키지가 그렇다고 생각하면 안 된다.

## 이번 장에서 일부러 생략한 것

이번 장에서는 metadata를 읽기만 했다. 아직 아래 문제는 뒤로 미뤘다.

- child dependency를 실제로 따라가는 방법
- 여러 버전 후보 중 하나를 고르는 문제
- peer dependency나 optional dependency 같은 추가 규칙

지금 필요한 건 "설치기가 registry 설명서에서 무엇을 꺼내야 하는가"를 분명하게 보는 일이다.

## 정리

이번 장의 핵심은 간단하다. 설치기는 이름과 버전만으로는 움직이지 못하고, registry metadata에서 tarball과 integrity와 child dependency 정보를 꺼내야 한다.

여기까지 읽은 뒤라면 아래 문장을 자기 말로 설명할 수 있어야 한다.

- metadata는 tarball URL과 integrity를 알려 준다.
- exact version만 지원하면 `versions[version]`을 바로 읽을 수 있다.
- 설치기는 registry JSON 전체를 들고 다니지 않고, 필요한 필드만 자기 구조로 정리한다.
