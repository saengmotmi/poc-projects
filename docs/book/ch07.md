# Chapter 7. 설치 결과를 기록하는 첫 lockfile

전이 의존성까지 따라가고 나면 설치기는 눈에 띄게 그럴듯해진다. direct dependency만이 아니라 child dependency까지 설치하고, 중첩 `node_modules`도 만든다. 그런데 이제 질문이 바뀐다. 오늘 성공한 설치 결과를 내일도 같은 모양으로 다시 꺼내 쓰려면 무엇을 남겨야 할까.

이때 필요한 것이 lockfile이다. 앞에서는 npm의 `package-lock.json`을 읽었고, 지금은 `pocpm` 자신의 lockfile을 만들어야 한다. 중요한 건 이름이 아니라 계약이다. 입력은 느슨할 수 있지만, 설치 결과는 구체적이어야 한다. 그 구체성을 기록하는 파일이 lockfile이다.

## 출발점에서 이미 되는 것

여기까지 구현된 `pocpm`은 적어도 아래를 할 수 있다고 본다.

- direct dependency를 읽는다.
- metadata에서 `dist`와 `dependencies`를 꺼낸다.
- 전이 의존성을 따라가며 패키지를 설치한다.
- nested `node_modules`를 만든다.

이제 해야 할 일은 설치 결과를 디스크 위의 폴더 구조만으로 끝내지 않고, 다음 설치를 위해 다시 읽을 수 있는 데이터 구조로 남기는 일이다.

## 이번 실험의 핵심 대상

이번 장에서 붙잡을 것은 두 가지다.

- resolution snapshot
- `pocpm.lock.json`

resolution snapshot이라는 말이 낯설 수 있는데, 감각은 단순하다. "이번 설치에서 실제로 무엇을 선택했는가"를 데이터로 고정한 결과라고 보면 된다.

## 왜 설치 결과를 따로 기록해야 할까

설치가 성공했다는 사실만으로는 충분하지 않다. 디스크에 폴더가 생겼다고 해서, 그 선택 과정이 자동으로 보존되지는 않는다. 다음 설치 때는 또다시 resolve를 해야 하고, 그때 같은 결과가 나온다는 보장도 없다.

그래서 lockfile은 "설치가 끝난 뒤의 디스크 상태"를 그대로 복사하는 파일이 아니라, "그 디스크 상태를 다시 만들 수 있도록 선택 결과를 요약한 파일"에 가깝다.

이 차이가 중요하다. 예를 들어 `registry-lib@1.0.0`이 `registry-leaf@1.0.0`로 풀렸다는 사실을 알고 있어야, 다음에도 같은 트리를 더 쉽게 재현할 수 있다.

## 현재 `pocpm`의 lockfile 쓰기 코드는 아주 작다

지금 `pocpm`의 [lockfile.ts](/Users/ohjongtaek/Desktop/dev/poc-projects/packages/pocpm/src/lockfile.ts)는 일부러 아주 작다.

```ts
export async function writeLockfile(project: ProjectContext, resolution: ResolutionResult): Promise<void> {
  await writeFile(project.lockfilePath, JSON.stringify(resolution, null, 2) + "\n");
}
```

처음 보면 너무 단순해 보여서 허무할 수 있다. 그런데 오히려 그 점이 중요하다. lockfile의 본질은 "얼마나 복잡한 포맷인가"보다 "무엇을 source of truth로 삼느냐"에 더 가깝다. 현재 `pocpm`은 `ResolutionResult` 자체를 lockfile의 원본 데이터로 삼는다.

즉 설치기가 resolve 단계에서 만들어 낸 결과를, 별도 변환 없이 그대로 기록하는 셈이다.

## `ResolutionResult`는 무엇을 담고 있을까

[types.ts](/Users/ohjongtaek/Desktop/dev/poc-projects/packages/pocpm/src/types.ts)를 보면 `ResolutionResult`는 아래처럼 생겼다.

```ts
export type ResolutionResult = {
  version: 1;
  resolutions: Record<string, string>;
  packages: Record<string, ResolvedPackage>;
};
```

핵심은 두 맵이다.

- `resolutions`
  - descriptor를 locator로 연결한다.
  - 예: `registry-lib@1.0.0 -> registry-lib@npm:1.0.0`
- `packages`
  - locator별 실제 패키지 정보를 기록한다.
  - 예: 버전, dependency 목록, dist 정보

이 구조를 눈으로 보면 lockfile이 왜 필요한지도 더 분명해진다. direct dependency 이름만 기록하는 것으로는 부족하고, "어떤 descriptor가 어떤 실제 패키지로 풀렸는가"와 "그 패키지는 어떤 정보를 갖고 있는가"가 같이 남아야 한다.

## 실제 `pocpm.lock.json`은 어떻게 생길까

fixture 프로젝트를 실제로 설치한 뒤 생성된 `pocpm.lock.json`의 핵심 부분은 아래와 같았다.

```json
{
  "version": 1,
  "resolutions": {
    "registry-bin@1.0.0": "registry-bin@npm:1.0.0",
    "registry-leaf@1.0.0": "registry-leaf@npm:1.0.0",
    "registry-lib@1.0.0": "registry-lib@npm:1.0.0"
  },
  "packages": {
    "registry-lib@npm:1.0.0": {
      "locatorKey": "registry-lib@npm:1.0.0",
      "name": "registry-lib",
      "version": "1.0.0",
      "reference": "npm:1.0.0",
      "source": "npm",
      "dependencies": {
        "registry-leaf": "1.0.0"
      },
      "bin": {},
      "scripts": {},
      "dist": {
        "tarball": "http://127.0.0.1:<port>/tarballs/registry-lib/1.0.0",
        "integrity": ""
      }
    }
  }
}
```

여기서 `<port>`는 mock registry가 뜰 때마다 달라지는 임시 포트다. 중요한 건 숫자 자체가 아니라, lockfile이 tarball 위치까지 같이 들고 있다는 점이다.

이 한 조각만 봐도 lockfile의 역할이 꽤 선명하다.

- `registry-lib@1.0.0`이라는 descriptor는
- `registry-lib@npm:1.0.0`이라는 locator로 풀렸고
- 그 패키지는 `registry-leaf@1.0.0`을 dependency로 갖고 있다

즉 lockfile은 단순한 버전 목록이 아니라, resolve된 패키지 그래프의 스냅샷이다.

## 왜 descriptor와 locator를 나눌까

처음 보면 `registry-lib@1.0.0`과 `registry-lib@npm:1.0.0`이 비슷해 보여서 두 키가 왜 필요한지 헷갈릴 수 있다. 여기서 두 층을 나눠 보면 이해가 쉽다.

- descriptor
  - "무엇을 원하느냐"
  - 예: `registry-lib@1.0.0`
- locator
  - "실제로 무엇으로 풀렸느냐"
  - 예: `registry-lib@npm:1.0.0`

지금은 exact version만 지원하므로 둘 차이가 작아 보인다. 하지만 구조를 나눠 두는 이유는 분명하다. 나중에 range, workspace, 다른 source가 생기면 "원하는 것"과 "실제로 고른 것"이 더 분명히 갈라지기 때문이다.

즉 지금은 단순해 보여도, lockfile의 모양은 이미 뒤의 확장을 버틸 수 있게 잡혀 있다.

## 검증

설치가 끝난 뒤 아래를 확인한다.

```bash
test -f pocpm.lock.json
node -e "const lock=require('./pocpm.lock.json'); console.log(lock.version); console.log(lock.resolutions['registry-lib@1.0.0']);"
```

fixture 기준으로 두 번째 명령에서 핵심 출력은 아래와 같았다.

```text
1
registry-lib@npm:1.0.0
```

이 출력이 보이면 적어도 lockfile이 생성됐고, descriptor에서 locator로 이어지는 매핑이 기록되었다는 뜻이다.

## 흔한 오해

- lockfile은 `node_modules`를 텍스트로 복사한 파일이 아니다.
- lockfile은 단순한 "설치 완료 표시" 파일도 아니다.
- lockfile은 버전만 기록하면 끝나는 파일이 아니다. 적어도 현재 `pocpm`에서는 `resolutions`와 `packages`가 함께 있어야 한다.

## 이번 장에서 일부러 생략한 것

이번 장에서는 lockfile을 쓰기만 했다. 아직 아래 문제는 남아 있다.

- lockfile을 다음 설치에서 실제로 읽는 방법
- frozen install
- lockfile format migration

지금 필요한 건 "설치 결과를 어떤 형태로 기록할 것인가"를 먼저 고정하는 일이다.

## 정리

이번 장의 핵심은 이렇다. lockfile은 설치 결과를 다시 재현할 수 있도록, resolve 단계의 선택 결과를 스냅샷으로 남기는 파일이다.

여기까지 읽은 뒤라면 아래 문장을 자기 말로 설명할 수 있어야 한다.

- `resolutions`는 descriptor를 locator로 연결한다.
- `packages`는 locator별 실제 패키지 정보를 담는다.
- 현재 `pocpm`은 `ResolutionResult` 자체를 lockfile에 기록한다.
