# Research: pocpm MVP

Date: 2026-04-21 00:35 KST

## Normal vs Current

- Normal behavior: `@poc/pocpm` should install a fixture project that uses exact npm versions plus `workspace:*`, produce `.pocpm-store`, `pocpm.lock.json`, `node_modules`, and `.bin`, then run a root script through the installed dependency binary.
- Current behavior: `poc-projects` does not yet contain a package manager workspace, fixtures, or tests that validate this flow.

## Output Axes

- Input: root `package.json`, workspace manifests, optional registry URL, current filesystem state.
- State: existing repo already uses Yarn Berry workspaces and has `packages/*` available for internal tooling.
- Environment: Node `v20.19.0` is available and already used for Vite+ validation in this repo.
- Time: current branch is clean and tracks `origin/main`.

## Repo Observations

- Root workspaces currently include both `projects/*` and `packages/*`.
- Existing internal package only provides shared tsconfig files under `packages/tsconfig`.
- Existing scripts already fan out to any `@poc/*` package through root task wrappers.

Root package snapshot:

```json
{
  "workspaces": ["projects/*", "packages/*"],
  "scripts": {
    "build": "vp run --filter \"@poc/*\" build",
    "check": "vp run --filter \"@poc/*\" check",
    "test": "vp run --filter \"@poc/*\" test"
  }
}
```

## Yarn Reference Observations

From the official Yarn docs:

- `yarn install` user documentation describes four steps: `Resolution`, `Fetch`, `Link`, `Build`.
- Yarn architecture documentation centers the implementation around three install steps: `Resolution`, `Fetch`, `Link`.

This MVP should follow the three-step install model and leave build orchestration out of scope.

## MVP Scope

### Included

- CLI command: `install`
- CLI command: `run <script>`
- Root manifest parsing
- Workspace discovery for simple `packages/*` style globs
- Exact npm version resolution only
- `workspace:*` resolution
- Registry metadata fetch + tarball download
- Integrity verification when metadata provides an integrity field
- Local tarball cache under `.pocpm-store`
- Hoist-less `node_modules` layout
- `.bin` symlink creation on Unix-like systems
- JSON lockfile output

### Excluded

- Semver ranges other than exact versions
- Peer dependencies
- Hoisting / dedupe
- PnP
- Publish / auth / private registries
- Windows cmd shim generation
- Complex workspace globs beyond the simple `dir/*` shape

## TDD Shape

The most useful tests are end-to-end fixture tests:

1. install root project with exact npm dependency and transitive dependency
2. install `workspace:*` dependency as a symlink and install its nested dependencies
3. create `.bin` entries and run a root script that uses a dependency binary
4. persist a stable lockfile snapshot and cache tarballs locally

## First Divergence to Avoid

Trying to start with semver, peer dependencies, or hoisting would expand the search space too early and obscure the three-step install flow. The MVP should keep the solver trivial so the fetch/link behavior stays visible.
